## 〇、先校准标尺：本文比较的「顶级」是什么

在 huginn 当前 harness 下测不出「顶级」——medal 字段是死代码（`mlebench_huginn.py:543-556`，leaderboard.csv 为未拉取 LFS 指针），synthetic 数据天花板（0.64/0.76/0.78）低于真实 Kaggle 铜牌线（`02_归因_MLE轨迹.md` F1/F2）。因此本文的「顶级」以**公开可核验的 MLE-bench 奖牌率**为标尺：

| Agent（骨架+模型） | MLE-bench 成绩 | 来源 |
|---|---|---|
| AIDE + o1-preview | 75 赛全量 16.9% any-medal、9.4% gold（pass@1）；pass@6 约翻倍 | MLE-bench 论文 arXiv:2410.07095（ICLR 2025）Table 2；AIDE 论文 arXiv:2502.13138 |
| AIRAmcts + DeepSeek-R1 | Lite 47.7% medal（从 39.6% 提升） | arXiv:2507.02554 |
| MLE-STAR + Gemini-2.5-Pro | Lite 63.6-64% medal（对比基线 25.8%/36.6%） | arXiv:2506.15692（NeurIPS 2025） |

## ① 顶级 agent 方法论清单（按对分数的边际贡献排序，标注 huginn 状态）

排序依据：公开消融/对比实验中能读出的增益量级，而非直觉。huginn 状态分三级：✅ 已具备 / ⚠️ 半成品 / ❌ 缺失。

### 第 1 档：决定性机制（无此则奖牌无从谈起）

**M1. 迭代预算与时间结构：24 小时 × 数百次评估，而非 30 分钟 × 60 次调用**
- 顶级做法：MLE-bench 官方协议 24h/赛（arXiv:2410.07095）；AIDE 在 24h 内持续 refine 最优节点（arXiv:2502.13138）；AIRA 发现非贪心搜索策略要到 ~10-19 小时后才追平并超过贪心（arXiv:2507.02554）；KompeteAI 给 6h/赛（arXiv:2508.10177）。
- huginn 状态：❌。`mlebench_huginn.py:575-576` 默认 `timeout=1800, max_tool_calls=60`；spaceship run#2 结语文本 "The budget is exhausted."（`workspaces/mlebench/spaceship-titanic/_huginn_meta.json`）；三任务时间预算只用了 25-46%（`09_归因_上下文与预算.md` F3）。
- 边际贡献：MLE-bench 论文实测把运行时长拉到 100h 仍有增益（虽趋缓）（arXiv:2410.07095，经 emergentmind.com/topics/mle-bench-evaluation-suite 转述）；huginn 60 calls 量级下连集成阶段都进不去。

**M2. 解空间树搜索 + 节点级 fitness 记录（而非线性版本链）**
- 顶级做法：AIDE 把每次解作为树节点，硬编码树搜索（Draft/Debug/Improve/Memory 算子）按验证分数选节点精修，「repeatedly refines partial solutions over the entire 24-hour window… fixes coding bugs and suboptimal hyperparameters rather than abandoning failed solutions」（arXiv:2502.13138 §1）；MLE-bench 论文 Table 2 显示同模型下 AIDE 骨架 8.7% vs OpenHands 线性骨架 4.4%（GPT-4o）。AIRA 进一步证明：在算子升级后，搜索策略（MCTS/进化）才开始贡献增益（arXiv:2507.02554）。
- huginn 状态：❌。may-2022 是线性 6 版脚本（`baseline.py → lgb_v3.py`，12 分钟内顺序产出，`02_归因_MLE轨迹.md` F5），无父节点回溯、无每节点分数档案、无「回到最优节点继续精修」机制——版本 6 停只是因为预算/时间到，不是因为搜索收敛。

**M3. 提交有效性契约：valid submission 是奖牌的入场券**
- 顶级做法：MLE-bench 论文把 Made Submission → Valid Submission → Above Median → Medal 作为漏斗逐级报告（arXiv:2410.07095 Table 2：AIDE+o1-preview 98.4% made → 82.8% valid）；论文 Discussion 直言「agents often failed to create valid submissions, despite having access to the validation server… did not always use the validation server」——顶级 agent 的护城河之一就是把「本地校验+试提交」做成强制步骤。
- huginn 状态：⚠️。有 `kaggle_submit_tool`，但其校验只有「行数/列名/数据类型/缺失值」（`agent/huginn/tools/bench_infra/kaggle_tool.py:1-6` docstring 原文），不校验值域 vs 指标契约（AUC 任务应提交概率，提交列唯一值 ≤2 即违约）——正是 may-2022 丢 0.04 AUC 的同一缺口（`02_归因_MLE轨迹.md` F3）。

### 第 2 档：从「有效提交」到「奖牌线」的核心增益

**M4. 指标-提交对齐 + 指标对齐的 CV（同一把尺贯穿选优、早停、提交）**
- 顶级做法：Kaggle GM 第一纪律是「CV 与竞赛指标一致、与 leaderboard 相关」——「Does your CV correlate with the leaderboard? This is the foundation. If it doesn't, every decision you make is based on wrong feedback」（uselessai.in/from-baseline-to-top-10 playbook checklist #1，2026-05-07；同见 The Kaggle Book 第 2 章 validation 设计）。AIRA 的 fitness function 直接定义为「validation set performance, 5-fold CV」（arXiv:2507.02554 形式化定义，经 emergentmind.com 转述）。
- huginn 状态：❌（单点失误但致命）。may-2022 全部 6 版脚本以 accuracy 做 CV 选优、AUC 任务交硬标签 `(test_probs >= 0.5)`（`workspaces/mlebench/tabular-playground-series-may-2022/lgb_v3.py:105-109`；description 明确要求 probability，`description.md:33,37`）；s3e18 则正确交了概率——能力在、纪律缺（`02_归因_MLE轨迹.md` F3）。

**M5. 算子质量 > 搜索策略：精修算子的工程质量是瓶颈**
- 顶级做法：AIRA 的核心实验结论——用 AIDE 原算子时 MCTS/进化相对贪心几乎无增益；升级算子集（prompt-adaptive complexity 防过早过设计、scoped memory 按算子类型检索兄弟/祖先记忆、think tokens、Crossover 重组）后 Lite 奖牌率 39.6%→47.7%（arXiv:2507.02554）；且仅把 AIDE 搬进更完善的 AIRA-dojo 执行环境，奖牌率 35.2%→45.9%（+30% 相对增益）——执行环境本身（文件系统、缓存、防限流）就是分数。
- huginn 状态：⚠️。执行环境有 Rust sandbox 静默崩溃、code_tool `eval()` 误杀、bash 超时等摩擦（`audit_20260717/19_ML方法论与自进化诊断.md` P-3/P-4），Improve 算子是「重写整个脚本」而非「定点 patch 最优节点」。

**M6. 集成：多模型族 + OOF stacking + 种子平均**
- 顶级做法：MLE-STAR 设独立 ensemble planner agent，先产出多个候选解、再由 agent 自己提出集成策略并按验证分数迭代精修该策略；消融显示集成模块对 gold 率有实质贡献（arXiv:2506.15692 §3.3 + Table 4）。Kaggle GM 标准打法：tabular 至少 XGBoost+CatBoost+LightGBM 三族（各家对类别/正则处理不同而互补）、blend 权重用 Optuna TPE 而非等权、stacking 在 OOF 预测上训 meta-model「稳定给出小而可靠的提升」、多种子平均（Fertilizers 赛 100 种子 XGB 集成稳定优于单种子）与定参后 100% 数据重训（NVIDIA Kaggle Grandmasters Playbook，developer.nvidia.com，2025-09-18）。
- huginn 状态：⚠️。有 5 折模型平均（`lgb_v3.py:105-109` 对 5 个 fold 模型取 mean）与 RF/LGB 双模型对比（spaceship `_code_tool_script.py:55-93`），但对比后只选其一，无 stacking、无权重优化、无种子平均、无全数据重训步骤。

### 第 3 档：奖牌线之上的抛光与防线

**M7. 泄漏与数据使用防线（确定性 checker，不靠 LLM 自觉）**
- 顶级做法：MLE-STAR 设 A_leakage 与 A_data 两个专用检查模块，消融给出教科书级反例：无 checker 时 LLM 用目标变量做特征工程，validation 0.8188→0.8677 上升而 test 0.8033→0.7343 暴跌（arXiv:2506.15692 Table 7/8，NeurIPS 2025 proceedings 版）。Kaggle GM 侧的标准件：对抗验证（train/test 二分类 AUC>0.6 即分布漂移，相应特征要丢）、预处理一律 fold 内拟合（target encoding/scaler/imputer 不得在全量数据上先 fit）、伪标签必须按 fold 生成 k 套防自泄漏（NVIDIA Playbook §6 + uselessai.in playbook Step 2/3）。
- huginn 状态：❌（MLE 路径）/ ⚠️（RCB 路径自知而不修）。MLE 三跑未做对抗验证、无 target encoding（未触发泄漏纯属数据简单）；RCB Material_003 自己发现了 >0.95 fingerprint similarity 泄漏却「只描述不修复」（`_agent_output.jsonl:147-150`，`19_ML方法论与自进化诊断.md` §1）。

**M8. Ablation 引导的定向精修（每步改动须有归因证据）**
- 顶级做法：MLE-STAR 外环用 ablation agent 生成「禁用/修改特定代码块」的变体脚本并实测，找出对性能影响最大的 ML 组件（特征工程/模型/集成块），内环只精修该块，历史 ablation 摘要作为上下文（arXiv:2506.15692 §3.2）。
- huginn 状态：❌。s3e18 末段把预算花在 5 depth × 3 n_estimators × 2 目标 × 5 fold = 150 次 RF 网格拟合上，而此时分数已在天花板 0.001 内（`02_归因_MLE轨迹.md` F2/F7 评估：方向对但边际收益≈0）——典型的「无 ablation 归因就全局抛光」。

**M9. 外部知识检索注入初始解（不纯靠模型内部知识）**
- 顶级做法：MLE-STAR 第一步用 web 检索「该任务类型当前有效模型」形成初始解，论文将此列为相对「rely on inherent LLM knowledge」的两大约束突破之一（arXiv:2506.15692 §1）；ML-Master 2.0 用 407 个被排除在 benchmark 之外的 Kaggle 竞赛预填充 L3「先验智慧」缓存（arXiv:2606.05250 §II-F 转述）。
- huginn 状态：⚠️。`MLE_TOOL_FILTER` 含 `web_search_tool`（`mlebench_huginn.py:47-57`），但 synthetic 模式下用不上；真实模式下必须先解决「搜到历史方案=污染」的评测卫生问题（`audit_20260717/16` H6；KompeteAI 的做法是屏蔽 RAG 访问被测竞赛的日期信息，arXiv:2508.10177 §5.1）。

**M10. 解的多样性作为一等指标**
- 顶级做法：AIRA 系 ideation diversity 研究在 11,000+ 条轨迹上测得多样性与成绩 Pearson r=0.57；用受控实验压低多样性（sibling memory 反向使用）后奖牌率绝对下降 ~7pp，证明因果（arXiv:2511.15593，经 alphaxiv.org 音频摘要转述）。
- huginn 状态：⚠️。LR→RF→LGB 渐进有一定多样性，但 6 版全是同一 pipeline 骨架的局部变体（同一 CV、同一特征集微调），无「架构级」分歧（如 linear / tree / kernel / NN 四族并探）。

**M11. 最终节点选择策略：按验证取 top-k 而非单点最优**
- 顶级做法：AIRA 实测系统性过拟合验证集——若用 test 分数做 oracle 节点选择，奖牌率可再涨 9-13pp；工程上等价的可行做法是「按验证分数取 top-3 节点，以集成/复评方式收口」，可补回大部分 gap（arXiv:2507.02554，经 emergentmind.com 转述）。
- huginn 状态：❌。最终解 = 最后一个版本（`lgb_v3.py`），无任何跨节点选择逻辑；且其「验证分数」本身因 M4 错配而不可比。

**M12. 跨 run 记忆回流：蒸馏经验预填充下一场**
- 顶级做法：ML-Master 2.0 的 L3 Prior Wisdom 缓存跨任务持久、可检索、任务无关（arXiv:2606.05250 §II-F）；MARS / R&D-Agent 有 run 内 lesson pool（同文 §II-B）；HASTE（arXiv:2606.30911）把「知识轴」形式化为与搜索轴正交的层级技能积累。
- huginn 状态：❌（结构性空转）。seed_38 有人工复盘但复盘本身误诊（把硬标签退化诊成过拟合，`38_benchmark_evaluation_lessons.md:187-193` vs `02_归因_MLE轨迹.md` F4）；S7 产物 `stable_principles.jsonl` 在审计时不存在、`load_stable_principles()` 写了没人调、写入/读取路径不相交（`19_ML方法论与自进化诊断.md` §8；`00_综合归因报告.md` 四）。

### huginn 已具备项（确认清单，防止过度修复）

- ✅ 交互特征狩猎：may-2022 挖出 `f_0*f_27` XOR 结构、s3e18 挖出 `f_0*f_1`/`f_2*f_3`，正是数据生成规则（`02_归因_MLE轨迹.md` F2/④）。
- ✅ 规范 CV 骨架：StratifiedKFold(5, shuffle, seed) + early stopping + 合理正则（`lgb_v3.py` 等）。
- ✅ 折间模型平均：5 fold 模型 mean 提交（`lgb_v3.py:105-109`）。
- ✅ baseline→强模型渐进策略与 sanity 对照（spaceship 额外验证「仅 CryoSleep 规则」，`_code_tool_script.py:100-102`）。

结论：huginn 的「单点技术肌肉」（特征工程直觉、GBDT 使用）已达熟手档；差距全部在**方法论的基础设施层**——搜索结构、契约校验、集成体系、防线 checker、记忆回流。这正是「中上→顶级」的差距本质：不是知道更多技巧，而是把技巧制度化为不可绕过的管线。

## ② 「中上→顶级」的十条具体机制差异

每条均为可工程化条目：有顶级 agent/Kaggle GM 的公开实现为证，有 huginn 的具体落点。

**D1. submission-format 契约校验器（metric → 提交类型硬约束）**
顶级：MLE-bench 把 valid submission 作为独立漏斗指标；论文 Discussion 指出失败主因之一是 agent 不用 validation server（arXiv:2410.07095）。
工程化：校验器输入 = 竞赛 metric（从 `grade.py`/`config.yaml` 解析）+ submission.csv；规则如「AUC/logloss/MAP@k 类指标 → 提交列必须为非二值浮点（nunique > max(2, 0.01×n_rows)）」「accuracy/F1 类 → 整数标签且类别集合 ⊆ 训练集类别」；违约即拒绝交卷并自动生成修复 patch（概率任务把 `(p>=0.5)` 行替换为 `p`）。
huginn 落点：`agent/huginn/tools/bench_infra/kaggle_tool.py`（现仅验行数/列名/NaN，`:1-6`）升级为契约校验；本次直接损失案例 `workspaces/mlebench/tabular-playground-series-may-2022/lgb_v3.py:105-109`（≈0.04 AUC）。

**D2. 指标对齐的 CV 选优（fitness function 从 description/grade 机械解析，不许 LLM 自选）**
顶级：AIRA 形式化 fitness = 验证集 5-fold CV 性能（arXiv:2507.02554）；Kaggle GM checklist 第 1 条「CV 与 leaderboard 相关」（uselessai.in playbook）。
工程化：harness 在 run 开始时解析 `mle-bench/mlebench/competitions/<task>/grade.py` 的评分函数名 + description.md 的 metric 段，生成「CV scoring = <metric>；early stopping monitor = <metric>；提交类型 = probability/label」三件套注入 system prompt，并在 sanity gate 用正则静态扫描 agent 脚本中 `scoring=`/`metric=` 参数是否匹配。
huginn 落点：`mlebench_huginn.py:367-369`（当前只写了 spaceship 的提交格式）扩为逐竞赛注入；对应根因 `02_归因_MLE轨迹.md` F3 促成条件。

**D3. 解空间树 + 节点档案（code, val_score, parent, operator）持久化**
顶级：AIDE 树搜索（arXiv:2502.13138）；AIRA 搜索图 G_t=(V_t,E_t) 形式化（arXiv:2507.02554）。
工程化：每个候选解落盘为 `solutions/node_<id>.py` + `nodes.jsonl` 记录（code hash, CV 分数, 父节点, 算子, timestamp）；Improve 算子输入 = 当前最优节点的代码+其分数历史；Debug 输入 = 失败节点+stderr；预算耗尽时提交「验证最优节点」而非「最后节点」。
huginn 落点：新模块（如 `agent/huginn/bench/solution_tree.py`），替代 may-2022 式线性 6 版脚本（`02_归因_MLE轨迹.md` F5）；复用现有 `.checkpoint.sqlite` 基础设施。

**D4. 预算量级对齐协议 + 相位预算**
顶级：24h/赛（MLE-bench 官方）；6h/赛（KompeteAI）；AIRA 测得探索性策略 10-19h 后才兑现增益。
工程化：MLE 真实跑分提到 ≥500 calls / ≥6h；把预算分相位（EDA≤15%、模型族探索≤35%、精修≤35%、集成与收口≥15% 强制保留），收口相位不可被前期透支。
huginn 落点：`mlebench_huginn.py:575-576`（60/1800 → 500+/21600+）；phase-aware 预算通道当前是死代码（`agent/huginn/bench/orchestrator.py:171-173` 要求 research mode 才生效），接线或显式重写为 benchmark 模式可用；前置修复 = checkpoint 悬空 tool_calls 400 崩溃（`09_归因_上下文与预算.md` F2，预算放大后崩溃率单调上升）。

**D5. 模型 zoo 多样性 + OOF stacking + blend 权重优化作为默认收口相位**
顶级：MLE-STAR ensemble planner agent 迭代精修集成策略（arXiv:2506.15692 §3.3）；GM 最低配 XGB+CatBoost+LGBM 三族 + OOF meta-model + TPE 权重搜索（NVIDIA Playbook；uselessai.in checklist #4/#5/#9）。
工程化：收口相位模板 = 固定同一 CV split 训 ≥3 模型族 → 生成 OOF 矩阵 → logistic/linear meta-model → Optuna TPE 搜 blend 权重（50 trials 内）→ 与单模型 CV 对比，只在 blend CV ≥ 最佳单模型时采用。
huginn 落点：作为技能/模板注入（knowledge seed + prompt）；当前 RF/LGB 对比后弃一（spaceship `_code_tool_script.py:55-93`）即损失点。

**D6. 种子平均 + 定参后 100% 数据重训**
顶级：100 种子 XGB 集成稳定优于单种子（NVIDIA Playbook Fertilizers 案例）；「retrain on 100% of the data」是 GM 标准最后一步。
工程化：最终管线固定超参后，{3-10 种子 × K fold} 预测平均；再出一个「全量训练」变体与 CV 变体按 D11 的 top-k 策略二选一。
huginn 落点：同一收口模板；当前无任何种子维度（全部单种子，`lgb_v3.py` 等）。

**D7. 泄漏/数据使用确定性 checker 进 sanity gate**
顶级：MLE-STAR A_leakage + A_data（消融反例：validation↑、test 0.8033→0.7343，arXiv:2506.15692 Table 7/8）；GM 对抗验证 AUC>0.6 判漂移（uselessai.in Step 2）。
工程化：sanity gate 三检查——(a) 静态扫描：特征工程代码是否引用目标列/测试集统计量（test 的 mean/median 用于 fillna 等）；(b) 对抗验证脚本：train vs test 二分类，AUC>0.6 报警并要求处理；(c) 预处理拟合位置检查：target encoding/scaler 是否在 fold 外 fit。
huginn 落点：接入 MLE 完成判据（见 D10）；并修复 RCB 侧「发现泄漏只描述不修复」的行为缺口（`19_ML方法论与自进化诊断.md` §1，`_agent_output.jsonl:147-150`）。

**D8. Ablation 引导的定向精修算子（替代全局抛光）**
顶级：MLE-STAR 外环 ablation agent 实测各代码块贡献、内环只精修最大贡献块（arXiv:2506.15692 §3.2）。
工程化：Improve 算子前置一步「消融实验」：对当前最优解生成「去交互特征/换模型族/去正则」等 2-3 个变体并各跑一次 CV，把分数差作为下一步精修目标的依据；禁止在无消融证据时启动网格搜索（判据：当前 CV 距理论/经验天花板 <0.005 时网格搜索 ROI=0）。
huginn 落点：迭代策略 prompt + 算子；反例 = s3e18 的 150 次 RF 网格抛光（`_code_tool_script.py:41-55`，`02_归因_MLE轨迹.md` ④「方向对但边际收益≈0」）。

**D9. 本地评分即服务（grade-as-a-tool），每次提交前强制自评**
顶级：MLE-bench 提供 validation server 且论文点名「agent 有而不用」是失败主因（arXiv:2410.07095 Discussion）。
工程化：把 `mle-bench/mlebench/competitions/<task>/grade.py` 包装为 `grade_submission_tool`（输入 submission.csv 路径 + 本地可得的验证标签/代理集，输出 metric 值），完成判据要求「最后一次 grade 调用成功且分数 ≥ 当前 CV 的 0.98 倍（防提交管线错误）」；同时实现 AIRA 式 top-3 节点选择（D11）。
huginn 落点：新工具 + `MLE_TOOL_FILTER`（`mlebench_huginn.py:47-57`）加入；synthetic 模式可用生成器已知噪声规则算 oracle 上限做归一化（`02_归因_MLE轨迹.md` 建议 4 已提出 ceiling-normalized）。

**D10. 跨 run 记忆回流：蒸馏 playbook 注入下次 run（写读路径闭合）**
顶级：ML-Master 2.0 用 407 个非 benchmark 竞赛预填充持久 L3 智慧缓存（arXiv:2606.05250 §II-F 转述）；HASTE 层级技能积累（arXiv:2606.30911）。
工程化：每次 run 结束生成结构化 lesson（任务类型, metric, 关键决策, 分数 delta, 失败 checker 命中记录），LLM 蒸馏成「if 任务特征 X then 策略 Y」条目入库；下次 run 的 system prompt 强制注入 top-k 相关条目；**先修评测信号再开回流**（否则蒸馏出的是「LFS 指针→try/except 兜底 medal=none」这类制度化谬误，`38_benchmark_evaluation_lessons.md:243-245`）。
huginn 落点：写侧 `agent/huginn/agent/reflection.py:459`（store_stable_principle 已存在），读侧 `agent/huginn/memory/longterm.py:1107-1128`（load_stable_principles 已存在未接线），路径统一（`00_综合归因报告.md` 四：引擎写 `~/.huginn/logs/`、context_builder 读 `$HUGINN_CACHE_DIR`）；复盘 checklist 增加「CV vs test gap 诊断前先核对两边同度量」——本次 F4 误诊的直接修复。

> 附：贯穿性差异（不计入十条但放大一切）——执行环境可靠性。AIRA 测得仅换更完善的执行环境，同算法奖牌率 35.2%→45.9%（arXiv:2507.02554）。huginn 侧对应项：Rust sandbox 静默崩溃纳入回退、code_tool `eval()` 误杀修复、checkpoint 400 崩溃修复（`19_ML方法论与自进化诊断.md` §4/§6、`09_归因_上下文与预算.md` F2）。这不是「ML 方法论」，但它给所有十条机制打折。

## ③ 机制 → huginn 四大落点映射

### 落点 1：tool_filter（`mlebench_huginn.py:47-57` 等适配器 allowlist）

当前 9 工具（code/bash/file_read/glob/grep/web_search/subagent/plot/kaggle_submit）。
- 加 `grade_submission_tool`（D9）——本地评分即服务。
- 升级 `kaggle_submit_tool` 为契约校验器（D1），验收判据：对 may-2022 式 {0,1} 提交在 AUC 任务下必拒。
- 恢复数学工具（symbolic_math/unit/validate）入 MLE/SAB 过滤清单——当前数学工具被结构性摘除（`00_综合归因报告.md` 三.4，`sab_huginn.py:53-62` 同族）；对抗验证/伪标签/种子平均等计算应走 code_tool，无需新工具，但泄漏 checker（D7）适合做成确定性 lint 工具而非 prompt 叮嘱。
- 真实模式开 `web_search_tool` 时按 KompeteAI 方式屏蔽被测竞赛可检索信息（D9/M9 的评测卫生前提）。

### 落点 2：完成判据（`agent/huginn/bench/orchestrator.py:147-159` `_is_done`）

当前：deliverable glob 全齐 + calls ≥ 50% budget，或 calls 达上限。升级为「sanity-gated 收敛判据」：
1. D1 契约校验通过；
2. D9 本地 grade 成功且与 CV 自洽（|grade − CV| / CV < 5%，否则判提交管线 bug 强制修复一轮）；
3. D7 泄漏三检查通过；
4. D5 收口相位已执行（存在 OOF stacking/blend 尝试记录，或明确记录「blend CV 未超单模型故弃用」）；
5. 提交物 = 验证分数最优节点（D3/D11），非最后节点。
同时修既有 bug：`MLE_DELIVERABLES` 要求 `submission/*.py` 而 agent 脚本写在 workspace 根目录导致 missing() 永真（`orchestrator.py:73-76`，`02_归因_MLE轨迹.md` F6）——判据本身先要自洽。

### 落点 3：sanity gate（当前缺失，新建）

作为 `_is_done` 与交付之间的确定性层（非 LLM），按 D1/D7/D9 实现 + 复用综合报告共识项：自测全绿、loss 真降、无占位数值、报告-产物一致（`00_综合归因报告.md` 六.3/六.4）。所有 gate 命中记录落盘到 `_huginn_meta.json`（`09_归因_上下文与预算.md` 建议 7 的可观测性项），供 D10 记忆回流做信用分配——本次 F4 误诊（把硬标签退化诊成过拟合）若当时有「提交唯一值=2」的 gate 记录，信用分配不可能错。

### 落点 4：记忆回流（自进化回路）

- 先修评测信号（medal 死代码 `mlebench_huginn.py:543-556`、LFS 拉取、分位数方向 `:551-556`），再开回流——铁序（`00_综合归因报告.md` 六原则提醒）。
- 写侧：run 结束由 LLM 蒸馏结构化 lesson（任务类型/metric/checker 命中/分数 delta），写 `stable_principles.jsonl`；读侧：接线 `load_stable_principles()`（`agent/huginn/memory/longterm.py:1115-1128`）进 system prompt，统一写读路径。
- 预填充冷启动：把本报告 §① 的 M1-M12 与 GM checklist（CV-LB 相关、fold 内预处理、对抗验证、三族模型、TPE blend、种子平均、双提交策略）蒸馏为初始 playbook 条目入库——等价于 ML-Master 2.0 的 407 赛预填充（D10），但来源是公开方法论而非竞赛轨迹，无污染风险。
- 复盘 checklist 加「指标-提交对齐」项（`02_归因_MLE轨迹.md` 建议 6 已提出），防 F4 类信用分配错误再被固化。
