## ① 数据与方法

### 1.1 「模式」在 huginn 中的实际清单（先界定概念，否则归因无从谈起）

代码库中并存 **5 个互相独立的「模式/相位」机制**：

| # | 机制 | 取值 | 切换入口 | 切换副作用 |
|---|------|------|----------|-----------|
| M1 | exec_mode `agent.mode` | `tool_call`（默认）/ `code_act` | 用户手动设实例属性（core.py:288-291） | chat() 早返回进 `code_act_loop`（streaming.py:556-571） |
| M2 | user_mode `agent._mode` | `chat`（默认）/ `research` / `plan` | `set_mode()`（core.py:412-445） | **`self._agent_graph = None`（graph 作废重建）** + `_phase_manager.reset()` + `permission_config.plan_mode` 翻转 |
| M3 | ResearchPhase 相位机 | 7 相 + OPEN | LLM 输出 `[PHASE:X]` 标记 → `transition_phase()`（streaming.py:50, 84-98, 1230-1236）；或 phase_tool | **`self._agent_graph = None`** + 对话树 fork + 换 PHASE_PROMPTS 前缀 + **换 PHASE_TOOLS 工具过滤** |
| M4 | CognitiveStateMachine（CSM） | S0–S7（含 S3_SWITCH/S6_FEEDBACK/S7_SELF_MODIFY） | reflection 后置信号（reflection.py:234-271） | S3/S6 置 `_needs_compaction` flag → 下一轮触发压缩（streaming.py:911-924） |

## ② 核心发现（按证据强度排序）

### F1【实证·强】在产生分数的 benchmark 运行中，「模式切换」作为运行时事件几乎从未发生

- 四个根适配器（mlebench/sab/rcb/paperbench_huginn.py）**全量 grep 无任何 `set_mode()` 调用**，也不设 `agent.mode`——所有得分运行固定在 M1=`tool_call`、M2=`chat` 的默认组合（core.py:291 默认值；streaming.py:556 `getattr(self, "mode", "tool_call")`）。
- PaperBench all-in-one 运行残留的 1321 条 checkpoint 中 `[PHASE` 标记 **0 命中**（字节级检索），即 M3 相位自动切换在该运行中一次都没触发。
- 因此 audit 04 的 P1-1/P1-2（code_act exec 挂死/沙箱逃逸）对本次跑分属于**背景缺陷**——code_act 模式从未被启用；M2 切换的 graph 重建代价（system prompt 重烘焙 + 工具重绑定）也从未被支付。

**推论**：字面上的「模式切换开销」（状态丢失、上下文重建）**不是**本次跑分差的原因——因为切换根本没发生。真正的问题在 F2–F4：模式机制「名义存在、实际空转/半失效」，以另一种方式拖累跑分。

### F2【实证·强】phase-aware 预算通道 100% 死代码：所有 benchmark 运行被统一低帽提前终止，而非按相位分配预算

- 机制设计：`PHASE_BUDGETS`（phases.py:48-56）给 EXECUTION 300 calls、VALIDATION 100、LITERATURE 50，总计 ~530；`PhaseManager.transition()` 时更新 `proposed_budget`（phases.py:226）；`BenchmarkOrchestrator` 读出来传给 `chat(budget_override=...)`（bench/orchestrator.py:161-174, 188-189）。
- **断路点**：`orchestrator.py:171-173` —— `mode = getattr(self.agent, "_mode", "chat"); if mode != "research": return None`。而 F1 已证实适配器从不进 research mode → `budget_override` 恒为 `None` → 每轮回退到构造参数的 `max_tool_calls`。
- **后果（直接失分原因）**：所有运行的终止都由 `max_total_calls` 触发，而非任务完成或超时：
  - SAB task_1：`--max-tool-calls` 默认 **40**、timeout 默认 1200s（sab_huginn.py:362-363），实际运行 **353s** 即结束（`workspaces/sab/task_1/_huginn_meta.json` duration_seconds=353）——远早于超时，【强推断】死于 40 次调用上限；judge 评语「code is incomplete (truncated mid-function), missing the main execution block… does not save predictions」、`output: 0`（`workspaces/sab/task_1/_score.json`），而残留 `pred_results/clintox_test_pred.csv` 只有 5 行数据——典型的「预算耗尽时执行闭环未走完」。
  - MLE 三任务：默认 60 calls / 1800s（mlebench_huginn.py:575-576），实际 450s / 827s / 792s 结束（各 `_huginn_meta.json`），spaceship-titanic 的最终输出原文开头即 **"The budget is exhausted. Let me summarize what was accomplished."**——模型自己声明被预算截停。三任务全部 no medal。
- 设计意图（执行阶段给 300 calls）与实际生效（全程 40/60/100 均一帽）之间的落差，是模式层对跑分**最直接**的贡献：不是「切换」丢分，而是「该切换/该按相位供预算的机制从未接通」。

### F3【实证·强】反思驱动的自适应模式切换是一条类型不兼容的死路径

- 生产者：每次工具失败，`TaskReflector.reflect()` 置 `should_switch_mode=True, suggested_mode="discover"`（task_reflector.py:130-131；成功推进 plan 时 `"construct"`，:162）；构造场景失败再置 `"discover"`（:165-169）。
- 消费者：`reflection.py:311-324` 调 `self.set_mode(reflection.suggested_mode)`。
- **断裂**：`set_mode()` 只接受 `("chat", "research", "plan")`，否则 `raise ValueError`（core.py:419-420）——`"discover"/"construct"` 永远非法，异常被 `except (ValueError, AttributeError): logger.debug(...)` 静默吞掉（reflection.py:319-324）。
- **影响定性**：「感知失败→切换探索模式」的自适应闭环**自始未闭合过**。对跑分这是背景缺陷（反正切换本就没发生，见 F1）；但它与 F2 共同说明：模式层的自适应设计在 benchmark 路径上整体空转。讽刺的是，这个 bug 反而起了「保护作用」——若它生效，每次工具失败都会触发 `set_mode` → `_agent_graph=None` + `_phase_manager.reset()`（core.py:424-428），长 benchmark 任务会被反复重置相位与 graph，损害会更大（推断）。

### F4【实证·强】autoloop（自进化模式）功能性塌缩：18/18 轨迹从未越过 perceive 相

- `agent/.huginn/trajectories/` 全部 18 份轨迹的 `metadata.phases` 只有两种模式：`["perceive", "report"]`（6 份）或 `["perceive"×5, "report"]`（12 份）；`total_time` 1.5–11.1s。hypothesize / plan / execute / validate / learn **零出现**。
- 报告侧印证：`huginn_autoloop_report_loop_2f1d2a47.md`（"Test the effect of strain on band gap"，Total Time 1.6s）tutor 总结自认 "the loop only ran *perceive*. The *act* phase … didn't execute"；多份报告含「被阻断/缺」门控字样（grep 命中 3 份），与 `engine.py:851-859` 的「阶段转移被阻断: 缺 evidence」机制一致——PhaseGate 证据门槛把 perceive→hypothesize 卡死，loop 空转到 max_iterations 后直接 report。
- **对跑分的影响**：**无直接影响**——四个 benchmark 适配器均不走 autoloop（它们用 `BenchmarkOrchestrator` while 循环或 rcb_runner 的 3-step 循环）。但对用户「自我进化的 AI 科学家」目标，这是核心模式的完全失效，属背景缺陷中最严重的一条。

### F5【实证·中】CSM→compaction 触发链在多数 benchmark 路径上是开启的，每 flag 付一次压缩 LLM 调用，且压缩管线本身半失效

- `reflection.py:251-253`：CSM 进 S3_SWITCH/S6_FEEDBACK 时 `_needs_compaction = not os.environ.get("HUGINN_RCB_CSM_SUBSET")`。只有 `agent/huginn/cli/rcb_runner.py:38` 设置了该环境变量；四个根适配器均未设（grep 实证）→ 走根适配器的运行中，每次工具失败后 CSM 迁移都可能置 flag，下一轮 `chat()` 开头触发 `summarize_compact_messages`（streaming.py:911-924，日志原文 "mode-switch triggered compaction (CSM S3/S6)"）。
- 但该压缩只作用于「本轮新消息」（checkpointer 开启时 inputs 不含历史，audit 05 P1-4，`context_builder.py:466-467`），且摘要降级链空转（audit 05 P1-2，`reflection.py:82-91` `primary_model` 恒取空串）。
- **代价定性**：每次 flag = 一次额外 summarize LLM 调用 + 摘要信息被提升进上下文而历史照旧增长——开销真实、收益近零。量级为每轮一次额外调用，属「税」而非「灾难」；对跑分为次要与间接因素（推断：无逐运行日志可直接计数）。

### F6【实证·中】相位工具过滤是一个未被触发但真实存在的陷阱：code_tool 在所有非 OPEN 相位均被过滤掉

- `code_tool` 未声明 `profile`，回落 `_default_profile()` → `phases=frozenset()`（tools/base.py:196-205，注释自述「仅 OPEN 阶段可用」）；`_rebuild_phase_tools()`（phases.py:143-151）对非 OPEN 相位只收 `_CORE_TOOLS`（phases.py:109-121，含 bash_tool 但**不含 code_tool**）+ 显式声明该相位的工具 → code_tool 在 LITERATURE…REPORTING 六个相位全部缺席。
- 同时 `_effective_system_prompt()` 无条件在 system prompt 尾部教 LLM「You can request a phase transition by including [PHASE:TARGET_PHASE]」（context.py:53-59），benchmark prompt 又大量使用 "Phase 1 / Phase 2" 措辞诱导模型输出该标记；一旦命中，`transition_phase()` 作废 graph 并按相位过滤工具（streaming.py:88-89 + context.py:163-165）。
- PaperBench 残留中 0 命中（F1）说明该陷阱**本次未引爆**（MLE/SAB 无 checkpoint 可核实，标推断）；属高危潜伏缺陷而非本次失分原因。

### F7【实证·中】rcb_runner 的 3-step 循环与 BenchmarkOrchestrator 是第三、第四套「工作流模式」，step 间靠 thread_id 共享 checkpoint 续上下文，无 graph 重建——本身不丢状态，但兜底档只认 paperbench 目录结构

- `bench/orchestrator.py:228-233` 的 `_has_code_no_output` 硬编码检查 `submission/reproduce.sh`；SAB（交付物 `pred_*.py`/`pred_*.txt`）与 RCB（`report.md`/`figures/`）的「代码已写、从未执行」形态永远匹配不到执行兜底档，只能收到泛化的 `_triage_prompt`（"You decide the minimal viable version"）。
- 这直接对应 RCB judge 高频评语「只描述不执行 / 占位数值 / 承认省略但未尝试核心方法」（`ResearchClawBench/score_results.json`，如 Material_000 index 2 图像项 score=0："no validation accuracy, no discovery rate"）与 SAB 的 output=0——**执行闭环没人强制收口**。这是工作流层缺陷，与 F2 的预算断路叠加：预算提前耗尽 + 兜底不对症 = 「描述了但没跑」。

## ③ 根因链（现象 → 机制 → 代码位置）

**链 A：SAB 25 分（代码截断/无执行/无预测文件）**
现象：353s 提前收工，output=0 → 机制：40 次调用的均一预算帽耗尽（设计中的 EXECUTION 相 300 calls 预算从未接通）+ 执行兜底档不认识 SAB 目录结构 → 代码：`bench/orchestrator.py:171-173`（mode!=research 即弃 budget_override）、`sab_huginn.py:363`（默认 40）、`bench/orchestrator.py:147-159`（_is_done 唯 budget/交付物二判）、`bench/orchestrator.py:228-233`（_has_code_no_output 硬编码 paperbench 布局）。

**链 B：MLE 三任务 no medal**
现象：模型自述 "The budget is exhausted"（spaceship `_huginn_meta.json` final_output_preview）→ 机制：同链 A 的预算断路（60 calls 帽），模型被迫在 baseline 阶段收尾，无迭代空间 → 代码：同上 + `mlebench_huginn.py:576`。模式切换本身无涉（0 次切换，F1）。

**链 C：RCB「只描述不执行/占位数值」**
现象：judge 判定报告只描述预期结果、无真实训练指标 → 机制：执行无人强制（兜底不对症，F7）+ CSM/compaction 每 flag 一次空转压缩税（F5）+ 预算均一帽（F2）；「反思应切探索模式」死路径使失败后的策略纠偏从未发生（F3）→ 代码：`task_reflector.py:130-131` ↔ `core.py:419-420` ↔ `reflection.py:319-324`；`reflection.py:251-253` → `streaming.py:911-924`。

**链 D：autoloop 自进化空转（非跑分项，但直接关系用户目标）**
现象：18/18 轨迹 perceive→report，1.5–11s → 机制：PhaseGate 证据门槛阻断 perceive→hypothesize 且无人 override，loop 空转到底 → 代码：`autoloop/engine.py:816-895`（_evaluate_transition 阻断分支）、轨迹 `metadata.phases` 实证。

**链 E（未引爆的雷）**：任一运行中模型若输出 `[PHASE:X]` → graph 重建 + code_tool 被过滤 → 代码：`streaming.py:50,84-98,1230-1236`、`phases.py:143-151`、`tools/base.py:196-205`、`context.py:163-165`。本次 PaperBench 残留 0 命中（F6），但 prompt 层在主动邀请该行为（context.py:53-59）。

## ④ 对用户问题的回答

### Q1：模式切换在实际运行中的代价是什么？

分「设计上」与「实际上」：

- **设计上**（若切换发生）：M2/M3 切换 → `self._agent_graph = None`（core.py:424、streaming.py:88）→ 下一轮 `build_graph()` 全量重建：system prompt 重烘焙（含 KG/KB/persona/taste/STABLE_PRINCIPLES 全量重装配，context.py:36-143）+ 工具重绑定 + 可能的工具集变化（M3 的 PHASE_TOOLS 过滤，F6）。M2 进 research 还会 `_phase_manager.reset(LITERATURE)` 重置相位历史（core.py:426）。消息历史不丢（checkpoint 按 thread 续），所以**状态丢失主要在「相位/工具/graph」层，不在对话层**；M4 的 S3/S6 不重建 graph 但置压缩 flag（F5）。工具权限变化：M2=plan → 写工具强制 ASK（core.py:431-433）。
- **实际上**（本次跑分）：上述切换事件 ≈ 0 次（F1），实际支付的代价是三个「不切换的代价」：① 预算通道断路导致全程均一低帽（F2，直接失分）；② 每轮 CSM/compaction 空转税 + reflection/evolution/event-polling 后置处理（reflection.py:173-346 每个有工具结果的回合都跑，F5）；③ 自适应纠偏从未发生（F3）。

### Q2：哪些 benchmark 失败可明确归因到模式机制缺陷？

- **可明确归因（直接失分）**：SAB task_1 的提前终止（40-cap 而非 1200s 超时，353s 收工）→ output=0；MLE 的 budget-exhausted 收尾 → 无迭代无 medal。归因对象是「相位预算机制断路」（模式层组件），证据链 A/B。
- **部分归因（模式层 × 工作流层叠加）**：RCB 的低分——执行兜底不对症是主因（工作流），反思模式切换死路径使「失败后换策略」缺席是叠加因（模式层，F3），证据链 C。
- **不可归因**：repro 基准的 χ=1.0 vs 0.004、θD=6.6K vs 480K 是物理量纲/数值错误（bench_repro.log），与模式机制无关；PaperBench rubric 大量 0–50 分主因是复现完整度与模型能力，模式层仅贡献 F5 的税。

### Q3：相比 EvoScientist/Claude 的单循环+强工具风格，模式设计是优势还是负担？

**在当前实现完成度下是负担**。理由：① 5 套模式机制中，M1 未被 benchmark 使用、M2 从不被切换、M3 的预算通道断路（F2）且工具过滤是雷（F6）、M4 只产税不产智（F5）、M5 整体塌缩（F4）——**没有任何一套在得分路径上提供了可观测的正收益**；② 它们仍要收维护与运行税：4 套状态机并行且靠 ad-hoc 同步（audit 04 P1-6）、每回合 reflection/CSM/event-polling 后置链、prompt 里常驻的相位教学段（context.py:53-59 白占 token 还埋雷）；③ 对照组事实：得分相对最好的 repro（8/10）走的是最朴素的直接 LLM 调用路径（reproduce.log 显示单轮生成），绕开了全部模式机制。理念上「相位化科研流程」并非错误，但**单循环+强工具+显式工作流编排（如 BenchmarkOrchestrator 自己的 while 循环）已经覆盖了 90% 的收益**，模式层当前是那 10% 都没兑现还倒贴的部分。

### Q4：模式层对跑分影响的权重评估

**次因（间接），权重估计 15–25%**。

- 主因（归于其他专项）：执行闭环缺失与预算/编排失配（工作流层，含 F7 兜底不对症）、模型能力与数值正确性（repro 物理错误）、评测完整性问题（audit 16 确认的 judge 降级、标签暴露等）。
- 模式层的真实贡献是 F2（预算断路 → 提前终止，SAB/MLE 的直接失分之一）、F3（纠偏缺席）、F5（空转税）。**「模式切换」本身作为事件对跑分影响为 0（F1）**——若用户直觉是「切来切去把上下文切丢了」，证据不支持；真相是「模式机器从没真正转起来过，但它的空转、断路和潜伏陷阱仍在扣分」。
