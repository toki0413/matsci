## ③ 根因链（现象 → 机制 → 代码位置）

### 链 A：budget 硬墙 → "只描述不执行"（RCB M_000、MLE spaceship）

```
现象：agent 中途宣告 "Both tool budgets are exhausted"，未跑核心管线就写报告
机制：ToolCallBudget 是硬墙，record() 超限返回 False 后无任何恢复/扩容通道
位置：agent/huginn/agents/tool_budget.py:64-67（max_calls / max_per_tool 超限）
      agent/huginn/cli/rcb_runner.py:303-306（max_tool_calls=150, per_tool=50；
      注释自承 "提高 code_tool 预算治 'budget exhausted 写不了 report' 短板"——
      说明该失败是反复出现的已知模式）
加重因素：RCB M_000 前期大量预算被沙箱拦截消耗——agent 与 "The sandbox seems to
      block loading custom classes"（torch.load pickle 类）搏斗 10+ 轮才读到数据
失分路径：核心实验未执行 → 报告只能写预期值 → judge "placeholder results" → 权重 0.5 项 0 分
```

```
      bash_tool 的回退只在【抛异常】时触发，静默错误结果不触发回退

      （:121-123 注释自承："它在某些场景 (RDKit+sklearn GPR) 会静默崩溃返回空 stderr,

      agent/huginn/tools/adapter.py:464-466（空 error 归一为 "Unknown error"）
```

## ④ 对归因问题的回答

### Q1 跑分任务中工具层失败占比多少？

| 口径 | 分子/分母 | 占比 |
|---|---|---|
| 外部 benchmark 出分任务（8 单元） | 5/8 有日志级工具层直接归因（RCB×3 + MLE spaceship + PaperBench run 崩溃） | **62.5%**（SAB 存疑，若计入为 6/8=75%，见下） |
| 内部 physics bench 失败题 | 3/3 全部伴随搜索链全灭 | **100%** |
| 内部 repro bench 失败题 | 0/2（无任何工具报错，agent 15s 猜值） | 0% |
| 长任务轮次 | 10/30 轮失败，全部工具/运行时层 | **33.3%**（失败轮中 100% 工具归因） |

**总判断**：在可观测的失败中，工具层是外部 benchmark 的**第一大直接失分项**（约 2/3），且全部 3 个 RCB 低分任务 100% 有工具层硬失败背书。

### Q2 哪些失败直接导致任务失分？

已建立「工具失败 → judge 扣分点」一一对应的有 5 条（详见 F1/F2/F5）：
1. budget 耗尽 → M_000 权重 0.5 项 0 分（placeholder results）
2. Rust sandbox 崩溃 → M_003 两项 0 分（无 BO/VAE/LOOCV）
3. circuit_open → M_001 权重 0.3 项 0 分（方法降级为 GP）
4. budget 耗尽 → spaceship-titanic 0.638 no medal
5. tool_calls 配对崩溃 → PaperBench run 夭折

**SAB task_1（25/100）标注存疑**：`pred_results/clintox_test_pred.csv` 仅 6 行（含表头，5 条预测）vs Clintox 测试集千级规模，output 维度 0 分与"未完整执行"一致；但 judge 评语 "truncated mid-function, missing the main execution block" 与当前 303 行完整文件（含 `if __name__ == "__main__": main()` 与保存逻辑，`_score.json pred_lines: 303`）**矛盾**——「推断」judge 评审了过程版本或评语失真，评语可靠性归 16 号审计；工具视角下 duration 仅 353s，budget/时间耗尽致中途收尾的可能性高【推断】。

### Q3 Windows 环境特有失败占多少？

- **确定 Windows 特有**：长任务 4 次 WinError 2 = 该轮次失败的 4/10 = **40%**；RCB M_003 的 Rust sandbox 静默崩溃（`huginn_ext` 原生扩展，开发者注释指向 "fork/exec 或内存限制"——Windows 无 fork 语义）1 例。
- **Windows 特征加重**：`.mcp.json` 的 `npx` spawn（npx.cmd 兼容问题，候选来源）；`reproduce.log:150-162` 中 agent 生成代码含 `plt.show()` 在无头 PowerShell 环境挂起 120s 超时。
- **平台无关**：budget 耗尽（链 A）、circuit_open（链 C）、搜索链全灭（链 E）、消息配对崩溃（链 F）在任何平台都会发生。
- **结论**：Windows 特有失败约占长任务失败的 40%、约占全部工具层失败事件的 1/4；但**平台无关的设计缺陷（budget 硬墙、静默失败不回退、熔断误触发、截断坏配对）造成的失分远大于 Windows 特有问题**。

### Q4 工具失败后的恢复行为是否存在？

| 场景 | 恢复机制 | 实际表现 |
|---|---|---|
| web_search 全灭 | 多级降级链 | **部分存在但无效**：4 级降级全部失败后无熔断信号给 agent，同题内死循环重试（链 E） |
| Rust sandbox 崩溃 | except 回退 SandboxExecutor（bash_tool.py:160-162） | **完全失效**：回退只捕获异常，静默错误结果不触发——这是"恢复机制名义存在"的典型 |
| circuit_open | half-open 探测 | **存在但代价高**：M_001 后期 code_tool 恢复，已耗预算 |
| budget 耗尽 | 无 | **硬墙**：agent 唯一选择是放弃执行、改写报告（链 A 两次重演） |
| 长任务 190s 超时 | checkpoint + resume_hint（routes/agents.py:208-221） | **部分有效**：任务6 R1-R4 连败后 R5 接续成功——但简单重试 4 次才成，恢复策略低效 |
| WinError 2 | 无 | 整轮作废 |

**总判断**：恢复行为"设计上有、实际上半残"。三种最致命的失败（budget 墙、静默崩溃、WinError 2）恰好都没有有效恢复路径。

### 关于「工作流 / 模式切换」的专门回答

跑分差的工具层根因不是工作流或模式切换本身，而是**"评测模式"与"默认模式"的分裂**：

1. `rcb_runner.py:30-46` 用 7 个 `os.environ.setdefault` 补丁（`HUGINN_ALLOW_LOCAL_BASH`、`HUGINN_NO_RUST_SANDBOX`、`HUGINN_HEALTH_MONITOR=0`、`HUGINN_SKIP_LOOP_DETECTOR`、`HUGINN_RCB_CSM_SUBSET` 等）绕开已知工具 bug——**开发者知道这些 bug，但修复策略是评测时打补丁绕过，而非修根因**。默认配置（生产环境、PaperBench/MLE/SAB runner）下 bug 全部健在。
2. git 时序证明补丁是被跑分失败**倒逼**的（失败运行在前、补丁在后），且补丁后四次重跑仍全部中断（v3-v6 无 done 记录）——补丁不充分：code_tool security scanner 误拦标准 GVAE `eval()`（220105 run）、PDF 过大读不了（214159 run）、脚本执行 walltime 等瓶颈仍在。
3. 把工具缺陷转嫁 prompt：`rcb_runner.py:283-284` 教 agent 绕过 scanner 误报、`mlebench_huginn.py:400-403` 教 "BUDGET DISCIPLINE"——占用上下文、且依赖模型自觉，不可靠。
4. 长任务 190s 是 HTTP 层硬超时（`routes/agents.py:209`），与 agent 的 3600s 工具超时设计（audit_20260717/07 P1-1）完全不匹配——长任务在这种架构下注定批量超时。

## ⑤ 可操作建议（按投入产出比排序）

1. **【半天 · 直接挽回 RCB/MLE 失分】把 budget 从"硬墙"改为"可续"**：`ToolCallBudget` 超限时不终止，而是强制 agent 先落盘中间产物（已生成的 submission/report 草稿）再给一次"收尾预算"（如 10 次仅限 file_write/code_tool）。当前 budget 耗尽=白跑（M_000、spaceship 两例实证）。
2. **【半天 · 消灭链 B】bash_tool 静默失败触发回退**：`run_sandboxed` 返回"success=False 且 stdout/stderr/message 全空"时视为崩溃信号，自动回退 SandboxExecutor 路径重跑一次，而非把 "Unknown error" 丢给 agent。同时默认设 `HUGINN_NO_RUST_SANDBOX=1`（在根因修复前让 Rust sandbox 默认退出 benchmark 路径）。
3. **【1 天 · 消灭链 D/E】统一失败语义 + 失败信号显式化**：落实 audit 07 P2-8——adapter 内把全部异常归一为 `ToolResult(success=False, error=...)`；web_search 全灭时返回显式 `"search_unavailable: true"` 而非空结果，让 agent 立即切换策略（查本地知识/声明不可考）而非重试烧钱。Windows 侧把 `.mcp.json` 的 `npx` 改为 `npx.cmd` 或 spawn 时 `shell=True`/`shutil.which` 解析。
4. **【1 天 · 消灭链 F】压缩前后校验 tool_calls 配对**：在 SummarizationMiddleware 之后加一个 middleware：凡 AIMessage 含 tool_calls，其后 N 条必须含对应 ToolMessage，否则补占位 ToolMessage（"tool output truncated"）。这是 LangChain 生态已知坑，修复是机械性的。
5. **【1 天 · 消灭链 C】熔断器按 benchmark 场景标定**：`HUGINN_HEALTH_MONITOR=0` 不应只存在于 rcb_runner 补丁——把"高失败率只读场景不熔断"做成 bench 编排层（`bench/orchestrator.py`）的正式配置项，file_read_tool 这类只读工具永久豁免熔断。
6. **【2 天 · 消除模式分裂】把 7 个评测补丁下沉为默认配置或删除**：`rcb_runner.py:30-46` 的每个 setdefault 都对应一个未修的根因；逐项评估——能修根因的修根因（Rust sandbox、熔断误触发、循环检测误判），不能修的做成 `huginn.toml` 正式配置并写明代价。禁止"评测模式特供补丁"继续存在，否则 benchmark 测的永远是另一个系统。
7. **【中期】长任务架构对齐**：HTTP 190s 超时与 agent 3600s 工具超时二选一——要么长任务走 checkpoint 续跑协议（resume_hint 自动化，而非靠调用方人肉重试），要么工具超时收紧到 HTTP 预算内并把剩余预算显式告诉 agent。
