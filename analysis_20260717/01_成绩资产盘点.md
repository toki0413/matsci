## ② 核心发现（按证据强度排序）

### 发现 1（决定性）：SAB 的 25 分主要是评分器截断伪影，不是 agent 真实水平

judge 评语三条核心指控——"truncated mid-function"、"missing the main execution block"、"does not save predictions"（`workspaces/sab/task_1/_score.json:13`）——**对完整文件全部为假**：

- 实际提交 `pred_clintox_nn.py` 共 **13,425 字节 / 303 行**，结尾有完整的 `if __name__ == "__main__": main()` 与 `out_df.to_csv(OUTPUT_CSV)`（文件尾 25 行已核读）。
- 评分器只送审 `code[:8000]`（`sab_huginn.py:311`），即前 59.5%；截断点恰好落在训练循环中段（`fda_prob = torch.sigmoid` 处切断）——judge 看到的字面现象就是"函数中间截断、无主执行块、未保存预测"。
- 结果：output 维度 0/20、correctness 10/40，合计 25/100（`_score.json:7-12`）。**这是测量系统制造的低分**；但注意该指标本身非 SAB 原生 success_rate（`sab_huginn.py:8-11`），即使修复截断也不可与官方榜单对比。

### 发现 2（决定性）：MLE spaceship-titanic 得分低于单特征规则基线——60 次工具调用产出的模型不如一行 if

重算（直接读 `_private/test.csv` 与 `submission/submission.csv`）：

| 基线 | 准确率 |
|---|---|
| 多数类规则 | 0.567 |
| **CryoSleep 单特征规则** | **0.642** |
| **huginn 提交**（450s、预算耗尽） | **0.638**（`workspaces/mlebench/spaceship-titanic/_score.json:3`） |

agent 自己的 EDA 已发现 "CryoSleep is the strongest single predictor (~80% Transported rate when True)"（`_huginn_meta.json:6`），最终提交却低于该单特征规则；meta 首句即 "The budget is exhausted"（同上）——**60 次工具调用预算（`mlebench_huginn.py:576`）在模型调优中途耗尽，且流程中没有"提交前 vs 平凡基线"的 sanity check**。

### 发现 3（强）：RCB 低分的直接机制是"早交卷"——轨迹 16~85 行、7~19 分钟就交付

- 53 个运行 workspace 仅 13 份有评分（完成率 25%）；Material_003 跑了 33 次仅 5 次有分（15%），分数 2.25~12.75 剧烈波动（`ResearchClawBench/workspaces/` 目录枚举 + 各 `_score.json`）。
- 批量 4 任务运行时长：Material_000=1117.7s、Material_001=**436.7s**、Material_002=794.4s、Material_003=**402.7s**（`batch_results.json`）——后两者只用掉 1800s 预算的 1/4 不到就结束。轨迹厚度：Material_003_160245 仅 **16 行** jsonl，Material_001 45 行、Material_002 51 行、Material_000 85 行。
- judge 高频评语与此互证："only describes expected or placeholder results"（`score_results.json:12`）、"no actual model performance metrics from running the pipeline"（同上）、"figures are not generated (only placeholder references)"（`:82`）、"admits these omissions but does not attempt the paper's core methodology"（`score_m003_v2.json:11`）。
- 机制在 prompt 层：硬编码的 PHASED PROTOCOL 强制 "Phase 3 (calls 31-40): WRITE report/report.md NOW … incomplete results are fine"，且 "DEEP LEARNING (VAE, transformers, GNNs) is FORBIDDEN until report.md exists"（`rcb_huginn.py:140-156`）——**为防"过度工程"而设计的阶段协议，把 agent 训练成了"先交卷、后补做"，而 judge 按论文核心方法保真度打分，两者目标错位**。这是工作流（编排协议）层直接失分，非模式切换逻辑缺陷。
- 另：deliverable 完成判据路径不一致（orchestrator 查根目录 `report.md`，实际要求 `report/report.md`），"交付齐全"分支永不触发（审计16 P2-1，`16_评测体系完整性.md:231-237`）——完成信号失真，每次运行要么早退要么烧满预算。

### 发现 4（强）：PaperBench 三类维度呈"骨架像样、执行全灭"形态；且该成绩在开卷条件下取得

| 论文 | 加权总分 | Code Development | Code Execution | Result Analysis | 0/50/100 分布 |
|---|---|---|---|---|---|
| all-in-one（174叶） | **13.52%**（另有 M7=12.36、T20=15.33 两次重评） | 26.7% | **0.8%** | **0.0%** | 144/12/17 |
| pinn（1963叶） | **2.23%** | 18.0% | **1.2%** | **0.0%** | 1887/53/19 |
| stochastic-interpolants（69叶） | **33.85%** | 38.2% | **0.0%** | **0.0%** | 47/4/18 |

（由 `workspaces/paperbench/*/_score*.json` 逐叶重算，与 declared total 一致。）

- 三篇平均 ≈16.5%，低于 Claude 3.5 Sonnet 21.0%、o1 26.6%、人类 41.4%（`baselines.py:34-36`）。
- **这是"开卷还考砸"**：评分 rubric 与具体叶节点答案被直接写进 agent prompt（审计16 P0-1，`paperbench_huginn.py:203-208/271-272/296-304`），judge 侧还有抬分引导（"If you can't tell, score 30"、"score 50+"，P1-6），harness 甚至代跑训练脚本补救 Execution（P1-1，`paperbench_huginn.py:401-498`）——三重加持下 Execution/Result 仍≈0%。**根因不在"信息不足"，在执行能力：agent 无法把代码真正跑通并产出可验证结果。**
- pinn 的 1963 个叶节点中 1815 个是 Code Execution 类（权重 2319/2514）——PaperBench 的计分结构决定了"只写代码不执行"必然崩盘。

### 发现 5（强）：内部 physics bench 失败与 web 检索全灭直接相关；repro 出现数量级/符号级物理错误

- `bench_run.log`（2026-07-14，**文件在 physics 段中途截断，无总结行**）：general 7/10 PASS（`:14-15`）；physics 可见 9 题中 6 PASS / 3 FAIL，第 10 题（Ni 内聚能）日志截断无判定。3 个 FAIL 全部伴随检索故障：`tio2_phase_stability` ΔE=0.0 vs 期望 -0.06（`:108`）、`cu_conductivity` 300 vs 期望 5.96×10⁷（**差 5 个数量级**，`:134`）、`pt_oxygen_adsorption` +1.1 vs 期望 -3.9（**符号错误**，`:250`）。日志同期 `duckduckgo_search` 包改名告警 + `urllib 兜底搜索也失败了: <urlopen error timed out>` 反复出现（`:28-31` 等 20+ 处），`next-gen.materialsproject.org` 403（`:103`）。**机制：检索基础设施死亡（`agent/huginn/tools/web_search_tool.py:249` 依赖已改名包）→ agent 无外部数据 → 硬答出错。**general 段的 `materials-bulk-modulus` FAIL（`:4`）则是另一类型：数值算对（60 GPa）但被判 FAIL，属答案提取/判分问题（与审计16 P1-7 的判分器缺陷同类）。
- `bench_repro.log`：8/10 PASS，2 FAIL 均为数量级错误——χ=1.0 vs 0.004（`:8`）、θD=6.6K vs 480K（`:10`）。
- `bench_lr.log`：15 题只记录 11 题即中断（文件止于 `[11/15]`），已记录部分 4 PASS / 7 FAIL；关键词命中 0/3、1/5 居多——谱系/概念类输出覆盖面不足。
- 注意：内部判分器本身宽松（任意数字命中即过、答错保底 0.3，审计16 P1-7），**内部通过率是上限，实际能力只会更低**。

### 发现 6（中）：长任务与压测的失分全部表现为"超时/轮次耗尽"，而非逻辑错误

- `_long_task_results.json`（2026-06-26）：4/6 通过；失败的任务5（带噪声实验数据）5 轮仅 2 轮成功、任务6（理论-实验矛盾）5 轮仅 1 轮成功。
- 压测 v22（2026-06-28）：21/27=77.8%，**6 个失败全是 "agent invoke timed out after 180/300s"**（Q1×2、Q3×3、Q4×1）；v23 基础设施自检 39/39；v24 三十分钟长航 tool_success_rate=1.0、RSS 正常。
- 解读：基础设施（熔断、审计、provenance）是健康的；**单点 agent 调用的时延预算与任务复杂度不匹配**是压测层唯一失效模式。

### 发现 7（中）：HLE 从未真正运行；自进化闭环是玩具级空转，未接入 benchmark 反馈

- 全仓无 `hle_results.json`，`.cache/hle/` 为空缓存；审计16 P0-3 同样确认"推断 HLE 适配器尚未真正跑过"（`16_评测体系完整性.md:92`）。官方基线 Gemini 3 Flash 33.7%（`baselines.py:55`）无从对比。
- 18 份 trajectory 全是微缩 loop：最小的一份（`agent/.huginn/trajectories/loop_fe3d6d8e.json`）objective="Compute something that will fail"、2 个 span、**tool_calls=[] 空**、总时长 1.56s。16 份 autoloop 报告**无完全空文件**（最小 1791 字节，修正 Orchestrator 勘察口径），但最小报告（`agent/huginn_autoloop_report_loop_2f1d2a47.md`）记录的 loop 只跑了 perceive 一个相位、1.6s、无任何 act——**内容上等效空转**。自进化系统从未消费 benchmark 失败经验（如 PaperBench 失分叶），与"自我进化的 AI 科学家"目标之间是断开的。

### 发现 8（对照价值）：描述型任务得分显著高于执行型任务

RCB 同一评分体系内：Astronomy_003（文献综述型）42.0~46.5，Math_001 38.0，Physics_000 33.9，而执行密集型的 Material_000~003 仅 2.25~17.28、Chemistry_000 4.95。**huginn 的能力剖面是"写作强、执行弱"**——与 PaperBench 的 Code Dev 26.7~38.2% vs Execution 0~1.2% 完全同构。这是跨 benchmark 一致的指纹。

## ④ 根因链（现象 → 机制 → 代码位置）

**链 A：RCB「只描述不执行」**
现象：judge 评语 placeholder/未执行（`score_results.json:12,28,82`）→ 机制：轨迹 16~85 行、402~1118s 提前交卷；硬编码阶段协议强制 calls 31-40 必须交报告、深度学习被禁至报告存在 → 代码：`rcb_huginn.py:140-156`（PHASED PROTOCOL + MODEL COMPLEXITY CEILING）；deliverable 判据路径不一致使完成信号失效：`agent/huginn/bench/orchestrator.py:67-71` vs `rcb_huginn.py:122-125`（审计16 P2-1）。

**链 B：PaperBench Execution/Result ≈0%**
现象：三篇 Execution 0~1.2%、Result 0% → 机制：agent 无法把训练真正跑通（无数据/依赖/时间），harness fallback 代跑也仅"让 judge 有东西可看"（`paperbench_huginn.py:492` 注释自承）；judge 只见 15 文件×6000 字符（`:740`，P3-1）→ 代码：`paperbench_huginn.py:401-498`（`_execute_training_fallback`）、`:740`。深层：PaperBench 计分权重压在 Execution（pinn 2319/2514 权重），只写骨架必然崩盘。

**链 C：MLE 低于平凡基线**
现象：0.638 < CryoSleep 规则 0.642 → 机制：60 次工具预算在调优中途耗尽（meta: "The budget is exhausted"），无"提交前 vs 平凡基线"检查环节 → 代码：`mlebench_huginn.py:576`（默认 60）、`:400-403`（prompt 只说"budget runs out"才停，未要求基线对照）。

**链 D：SAB 25 分（测量伪影）**
现象：judge 称截断/无主块/未保存 → 机制：送审代码 `code[:8000]`，实际文件 13425B，后 40%（含 main 与 to_csv）不可见 → 代码：`sab_huginn.py:311`。**直接失分原因是评分器，不是 agent。**

**链 E：physics bench 数量级/符号错误**
现象：cu 导电率差 5 个数量级、Pt 吸附能符号错 → 机制：web 检索全灭（ddgs 改名 + bing 超时 + MP 403），agent 无外部常数硬答 → 代码：`agent/huginn/tools/web_search_tool.py:249`；证据 `bench_run.log:19-31, 89-103, 108, 134, 250`。

**链 F：repro/压测/长任务的超时失效**
现象：v22 六连败全是 timeout、bench_lr 中断、长任务轮次耗尽 → 机制：单点 agent invoke 180/300s 预算与任务复杂度不匹配 → 代码：压测超时配置（`agent/_stress_test_v22_matsci_depth.py` 系列）；背景缺陷：审计04 指出 `autoloop/engine.py` 7185 行上帝类与 4 套状态机并存（`audit_20260717/04_架构_核心智能体循环.md:28`），调用链时延不可控。

**背景缺陷（存在但非本批任务直接失分原因）**：泄漏通道 P0-1/P0-2/P0-3/P1-2（利用未证实）；judge 与 agent 同源 P1-3；内部判分宽松 P1-7/P1-8；`_rescore_m7_c2st.py` 事后覆写 P1-4；medal 逻辑颠倒 P1-10（因 LFS 指针未爆发）。这些让分数"不可比"，但不解释"为什么低"——低分由链 A/B/C/E/F 解释。

## ⑤ 对用户问题的回答（工作流 / 模式切换 / 其他）

**Q：跑分差是工作流问题还是模式切换问题？**

以证据回答：**主因是工作流（编排协议+预算+基础设施）层，占可解释失分的绝大部分；模式切换（orchestrator 分流逻辑）未见直接失分证据。**

1. **工作流层（主因，证据充分）**：
   - RCB 的 PHASED PROTOCOL 是硬编码在 prompt 里的"工作流"，它直接制造了早交卷（链 A）；
   - MLE 的 60 次调用预算是工作流参数，直接制造了低于平凡基线的提交（链 C）；
   - deliverable 路径不一致（P2-1）是 orchestrator 完成判据缺陷，使"何时算完成"这一工作流信号整体失真；
   - web 检索全灭（链 E）是工具层基础设施失效，属工程而非模式问题。
2. **模式切换层（未见直接失分）**：三档分流/模式判断本身在本批成绩中无可归因的失败案例；压测与长任务的失败形态是超时而非错路由。审计04/05 的批评（多框架沉积、降级链空转）是背景健康度问题，未在本批任务中起作用。
3. **其他两个被低估的因素**：
   - **测量层伪影**（SAB 截断、内部判分宽松、judge 同源）——它双向扭曲：SAB 被低估、内部 bench 被高估，导致"自我感觉 70-80 分、外部实测 10-30 分"的落差；
   - **自进化闭环未接入评测**：18 轨迹全是 1.6s 级玩具 loop（tool_calls 为空），16 份报告无实质迭代内容——benchmark 失败经验（如 PaperBench 174 叶的逐条失分原因）从未回流为能力改进。用户目标的"自我进化"目前是有报表、无进化。
