## ② 核心发现（按证据强度排序）

### F1【直接证据·强】SAB 25/100 的主因是「judge 视野截断」，不是 agent 上下文/预算问题

`workspaces/sab/task_1/_score.json` 的 breakdown：`correctness=10/40, output=0/20, quality=10/20, science=5/20`，reasoning 原文："The code is incomplete (truncated mid-function), missing the main execution block and the actual training loop. It does not save predictions to the required output file."

但提交物 `workspaces/sab/task_1/pred_clintox_nn.py` 实测 **10,880 字符 / 303 行**，文件完整：尾部含训练循环、`out_df.to_csv(OUTPUT_CSV, index=False)` 与 `if __name__ == "__main__": main()`；要求的产物 `pred_results/clintox_test_pred.csv` 也存在。文件 mtime（2026-07-15 02:17:56）早于 `_score.json`（02:18:36），排除评分后补写。

根因：`sab_huginn.py:311` 把提交代码 `code[:8000]` 后才喂给 judge。截断点恰在 `main()` 的 train/val split 处（实测第 7950-8100 字符），**训练循环、保存预测、`__main__` 块全部落在 judge 视野之外**。judge 的三条差评与 8000 字符截断面一一对应——judge 如实描述了它看到的半截文件。

- 失分归因：`output=0/20` 全部、`correctness` 大部分系测量误差。**估算影响 30-40 分（推断）**：若 judge 看到全文件，该任务合理得分约 55-65/100。
- 这属于评测完整性层（审计 16 辖区，但该截断点未被审计 16 记录，为新证据），**不能记到 agent 的上下文/预算头上**。agent 侧事实：40 次调用预算跑满（353s），交付物完整。

### F2【直接证据·强】PaperBench M1 run 在第 16/150 次调用处死于「上下文状态完整性」崩溃，整 run 报废

`_m1_full.log` 完整记录：run 以 `timeout=3600s, max_tool_calls=150` 启动（:6），正常推进到 `tool #16: code_tool`（:26），下一轮图调用即抛：

```
openai.BadRequestError: Error code: 400 - An assistant message with 'tool_calls'
must be followed by tool messages responding to each 'tool_call_id'.
```
（`_m1_full.log:268-270`， traceback 贯穿 `orchestrator.py:180 → streaming.py:1041-1071 → langgraph`）

即 checkpoint 状态中存在「悬空的 AI tool_calls 消息」（其后缺对应 ToolMessage），整段历史再发给 API 时必然 400。streaming 重试分类（`streaming.py:1138-1143`）只把 rate_limit/overloaded/network/overflow 列为 retryable，400 invalid_request **不重试、不修复状态、不回滚 checkpoint** → 整个 run 直接死亡，剩余 134 次调用预算全部浪费。`_m1_smoke4.log` 另有一例退化 run：`Agent finished. Tool calls: 0, turns: 0`。

机制（推断，两种候选）：① 上一轮在 `max_tool_calls`/`recursion_limit` 切断图执行时，AI 已发出 tool_calls 而 ToolMessage 尚未落 checkpoint；② LLM 并行发出多个 tool_calls 时部分被预算/路由层拦截且未补占位 ToolMessage。两候选的共同根源都在审计 05 P1-4：checkpoint 状态只增不修、无任何 `RemoveMessage`/`update_state`/`trim_messages` 调用（全库 grep 为零，审计已确认）。

### F3【直接证据·中】MLE 三个任务全部被「工具调用预算」卡死，时间预算只用了 25-46%

`workspaces/mlebench/spaceship-titanic/_huginn_meta.json`：`duration_seconds: 450`，agent 结语文本原文 "The budget is exhausted."；defaults 为 `timeout=1800, max_tool_calls=60`（`mlebench_huginn.py:575-576`）。三个任务耗时 450s / 792s / 827s = 时间预算的 25% / 44% / 46%，即 **60 次调用在 7.5-13.8 s/call 内耗尽，wall-clock 大量闲置**。 medals 判定为 leaderboard 分位数（`mlebench_huginn.py:551-556`，bronze=中位数），0.638/0.745/0.758 均在中位以下。

归因：这是四处 benchmark 中**唯一确认「预算直接绑死」**的案例。但注意 agent 已发现关键特征（f_0*f_27 交互，CV AUC 0.83，见 `_huginn_meta.json` 预览），瓶颈在迭代次数而非上下文。预算翻倍的边际收益真实存在但有限（推断：+0.02-0.05 分，medal 仍不确定）。

### F4【直接证据·中】RCB「描述代替执行」发生时预算大量剩余——不是预算/上下文问题

`ResearchClawBench/batch_results.json`：四个任务耗时 1117.7s / 436.7s / 794.4s / 402.7s，全部 `status=completed, exit_code=0`；实际运行器 `agent/huginn/cli/rcb_runner.py:305` 给的是 `max_tool_calls=150`。judge 评语（`ResearchClawBench/score_results.json`）显示失分模式为「只描述不执行 / 占位数值 / 未尝试核心方法」：Material_003 idx1（w=0.3, score=0）"does not perform Bayesian optimization in the VAE latent space, does not generate novel vitrimer chemistries"；Material_000 idx2（w=0.5, score=0）"provides only qualitative descriptions... no validation accuracy"；Material_003 idx0 "does not implement a graph VAE"。

Material_003 只用 402.7s 就「完成」——agent 自己停了，不是被杀。两个 workflow 设计事实促成了「描述代替执行」：
- `rcb_runner.py:291` system prompt 明确要求 "Write report/report.md EARLY, then OVERWRITE as you add results"——报告先行，执行后置；
- `rcb_runner.py:321-326` 注释自述用「3 步认知循环」（提取→执行→自验证）替代完整流程，Step 3 后 agent 自然收尾。

**预算分配不当（workflow 层）有部分责任，但「预算量/上下文耗尽」无责任**（推断：归因占比 0-10%）。另注：`rcb_huginn.py:122-125` prompt 要求产物在 `report/report.md`、`report/images/`，而 `orchestrator.py:67-71` 的 `RCB_DELIVERABLES` 检查根目录 `report.md`、`figures/*.png`、`data/*.csv`——交付物定义与 prompt 错位（对本次被评分的 rcb_runner 路径无直接影响，但同族 bug）。

### F5【确认事实·中】PaperBench 预算是官方量级的 1/12（时间）/约 1/3-1/5（步数），真实压住了 Execution/Result Analysis 叶节点

`workspaces/paperbench/all-in-one/_score.json`：total 13.52，174 叶分布 {0:144, 30:1, 50:12, 100:17}；按类：**Code Development n=92 avg=24.8；Code Execution n=62 avg=0.8；Result Analysis n=20 avg=0.0**。Execution+Analysis 共 82 叶（47%）几乎全灭。judge 评语证实缺口是「规模没做」而非「代码没写」：idx89 "only trains with a single n_train value (default 5000). There is no evidence... trained separately [for 10^3/10^4/10^5]"；idx53/58 "No Tree/HMM task implemented"。

对照：PaperBench 官方给 agent **12 小时**（arXiv:2504.01848），不设工具调用上限；按 1-3 min/步估算官方 agent 典型步数约 240-720 步（推断）。huginn 适配器为 `timeout=3600, max_tool_calls=150`（`paperbench_huginn.py:916-917`）——**时间 1/12、步数约 1/3-1/5，量级不足成立**。完整复现该论文需 4+ 任务 × 3 档模拟预算 × 3 种 attention mask 的真实训练，150 次调用不可能覆盖。

但必须同时记录反向证据（审计 16 已确认）：该适配器把 rubric 叶节点答案直接写进 agent prompt（`paperbench_huginn.py:296-304` "RUBRIC ALIGNMENT" 节，审计 16 判定泄漏）、harness fallback 代替 agent 执行训练（`paperbench_huginn.py:401-498`，注释直言 "让 judge 有东西可看 （否则 Code Execution 全 0)"）、regex 覆写 judge 给分（`paperbench_huginn.py:651-672`）。**13.52 分本身已被三重机制抬高**；官方同赛道 Claude-3.5-Sonnet 也只有 21.0%、GPT-4o 4.1%。预算加到官方量级能补上的是 Execution/Analysis 的「真实训练规模」，预估 +15~25 分（推断），但同时受模型能力天花板约束。

### F6【确认事实·背景缺陷】审计 05 的上下文层缺陷在本批运行中「潜伏」而非「发作」

逐项核对审计 05 的行号（全部属实），但对本批成绩的作用需要区分：

| 缺陷 | 位置（已复核） | 本批运行中的作用 |
|---|---|---|
| P1-4 compaction 只压本轮新消息、checkpoint 状态无限增长、无 RemoveMessage | `streaming.py:906`（inputs 只含本轮新消息）、`streaming.py:302-419`（_maybe_auto_compact 只 promote summary，after% 用 system_prompt+summary 重算，属装饰性指标）、`context_builder.py:466-467` | **间接发作**：F2 的 400 崩溃与「状态只增不修」同根。直接 token 溢出未在日志出现 |

### F7【直接证据·强】repro 基准的 2 个失败与预算/上下文零相关

`bench_repro.log:8,10`：repro-curie-weiss χ=1.0（期望 ~0.004）用 15.6s 完成；repro-debye-temp θD=6.6K（期望 ~480K）用 13.8s 完成。数量级级物理错误（单位/公式错误），运行正常结束，与预算、上下文无任何关系——纯能力/正确性问题。

## ③ 根因链（现象 → 机制 → 代码位置）

**链 A：SAB 低分（测量误差链，非上下文/预算）**
现象：judge 评语「truncated mid-function / missing main / no save predictions」→ 机制：judge 只收到提交物前 8000/10880 字符，训练循环与 `to_csv` 在视野外 → 位置：`sab_huginn.py:311`（`code[:8000]`）→ 失分：output 0/20 + correctness 大部分，约 30-40 分（推断）。

**链 B：PaperBench M1 整 run 报废（上下文状态完整性链）**
现象：tool #16 后下一轮即 400，run 死 → 机制：checkpoint 中残留悬空 `tool_calls` AI 消息（缺 ToolMessage），全量历史重放必现 400；重试分类不含 invalid_request，无状态修复/回滚 → 位置：崩溃面 `streaming.py:1133-1143`（`states_yielded > 0` 直接 re-raise + 400 非 retryable）；状态只增不修 `streaming.py:302-419`（compaction 不删消息）、审计 05 P1-4（全库无 `RemoveMessage`/`update_state`）；悬空对产生点（推断：预算/递归限制切断图 mid-flight，或并行 tool_calls 部分被拦截未补占位消息）→ 失分：整 run 134 次剩余调用作废；同类崩溃在长 run 中概率随状态增长单调上升。

**链 C：MLE no medal（调用预算绑死链）**
现象：450s/1800s 时 agent 自述 "The budget is exhausted" → 机制：`max_tool_calls=60` 先于 timeout 耗尽，`BenchmarkOrchestrator._is_done` 在 `calls >= max_total_calls` 处强制收尾 → 位置：`mlebench_huginn.py:575-576`、`orchestrator.py:154-155` → 失分：迭代次数不足，特征工程/集成/调参未展开；估计为 medal gap 的 30-50%（推断）。

**链 D：PaperBench Execution/Analysis 全灭（预算量级不足链 + 测量膨胀）**
现象：82/174 叶（47%）均分 <1 → 机制：完整复现需多任务×多模拟预算×多 mask 的真实训练，150 calls/3600s 只能覆盖单点训练（n_train=5000 一档）→ 位置：`paperbench_huginn.py:916-917`；对照官方 12h/不限调用 → 失分：推断 +15~25 分的上行空间被预算封住。注意反向：rubric 泄漏（`paperbench_huginn.py:296-304`）+ fallback 代跑（`:401-498`）+ regex 覆写（`:651-672`）已把 13.52 抬高，真实能力分更低（审计 16）。

**链 E：RCB 描述代替执行（workflow/预算分配链，非预算量）**
现象：402-1117s 即 completed，judge 差评「未尝试核心方法」→ 机制：prompt 要求 report EARLY（`rcb_runner.py:291`）+ 3 步循环（`:321-326`）引导早收尾；交付物检查与 prompt 路径错位（`orchestrator.py:67-71` vs `rcb_huginn.py:122-125`）→ 失分主因仍是方法选择/能力（未实现 GNN/VAE/BO），预算分配负次要责任（推断 0-10%）。

**链 F：上下文压缩管线整体空转（潜伏链）**
现象：无任何 compaction 生效证据 → 机制：`HUGINN_CONTEXT_BUDGET_TOKENS` 缺省 → `context_budget_tokens=0` → `streaming.py:911/932` 两分支全跳过；唯一防线 `_maybe_auto_compact`（60% 阈值）只 promote summary 不删消息，且其 after% 用 system_prompt+summary 重算（`streaming.py:366-375`）制造「压缩成功」假象 → 位置：`config.py:750-752`、`streaming.py:302-419, 906-967`、`context_builder.py:466-467` → 当前无直接失分，预算放大后将放大链 B 崩溃率。

## ④ 对专项问题的回答

**Q1：SAB 代码截断能否归因到上下文耗尽/压缩失效/预算分配不当？**
不能。agent 产物完整（303 行、含 `__main__` 与保存预测），40 次调用预算跑满且交付物落地。截断发生在 judge 侧 `code[:8000]`（`sab_huginn.py:311`）。这是评测完整性问题，30-40 分（推断）记到测量层，不记到上下文/预算层。

**Q2：PaperBench「没做完」能否归因到预算？**
部分能，且是双层的：① 预算量级确实不足——1h/150 calls vs 官方 12h/不限调用（约 1/12 时间、1/3-1/5 步数），直接封死 Execution+Result Analysis 共 82 叶（均分 0.8/0.0），推断 +15~25 分空间；② 上下文状态完整性缺陷造成 M1 整 run 在第 16 次调用处死亡（400 悬空 tool_calls），属确定性浪费。但须扣除：当前 13.52 分已被 rubric 泄漏/harness 代跑/regex 覆写抬高（审计 16），且官方最强 agent 也只有 21%——「没做完」是预算、能力、测量三方叠加，预算层解释其中约 1/4-1/3（推断）。

**Q3：RCB「描述代替执行」能否归因到预算/上下文？**
基本不能。四个 run 全部正常 completed，耗时 402-1117s 远低于预算，150 次调用未耗尽。失败模式是 workflow 诱导（report-EARLY prompt + 3 步循环）+ 方法选择/能力（GNN/VAE/BO 未尝试）。预算「分配」（phase/工作流设计）负 0-10% 责任（推断），预算「量」与上下文无责任。

**Q4：150 次工具调用对论文复现是否量级不足？**
成立。官方 PaperBench 不设调用上限、给 12h（典型 agent 估算 240-720 步，推断）；150 calls/3600s 约为官方量级的 1/5-1/12。对 MLE（60 calls，时间用 25-46%）与 SAB（40 calls，且 `SAB_DELIVERABLES` 要求任务从未要求的 `pred_*.txt`——`orchestrator.py:78-81`——导致 `_is_done` 永不满足、必然烧满 40 次）同样偏紧。RCB 的 150 次则不紧（没用完）。

**Q5：多少分丢在预算/上下文层？（定量估计，全部推断）**

| 基准 | 当前分 | 满分 | 丢分中可归因预算/上下文层 | 主要归属 |
|---|---|---|---|---|
| SAB t1 | 25 | 100 | ≈ 0-5 分 | 测量层 30-40 分；余为 judge 严格度/能力 |
| PaperBench | 13.52 | 100 | ≈ 20-30 分（预算量 15-25 + 状态崩溃 5-10，需先扣除测量膨胀） | 能力/优先级 + 测量完整性 |
| RCB（均 ~13.7） | ~13.7 | 100 | ≈ 0-9 分（预算分配/workflow） | 方法选择/能力为主 |
| MLE | medal=none | medal | medal gap 的 30-50%（调用预算绑死） | 模型能力/迭代质量 |
| repro | 8 | 10 | 0 分 | 物理正确性 |

**汇总（推断）：上下文+预算层合计解释四个外部基准总失分的约 10-20%，不是主因。** 主因排序：① 能力/方法选择（RCB 不尝试核心方法、repro 物理错误）；② 评测完整性（SAB judge 截断、PaperBench 三重膨胀，方向相反但都扭曲分数）；③ 才是预算/上下文层（MLE 绑死 + PB 量级不足 + 偶发整 run 崩溃）。

**预算-产出曲线估算（PaperBench，deepseek-chat，推断）**：150c/1h → 现状 13.5%（含膨胀）；300c/2h → 15-18%（部分 Execution 叶复活）；600c/6h → 17-21%（逼近官方 Claude-3.5-Sonnet 区间，但受 deepseek-chat 能力天花板压制，官方 DeepSeek-R1 同赛道仅 6.0%）；边际收益递减——第 N+1 档模拟预算×任务×mask 的组合训练每档要吃数十次调用与真实 GPU/CPU 小时。**前提：先修链 B/F，否则预算越大崩溃越多。**
