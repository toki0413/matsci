## ② 核心发现（按证据强度排序）

### D1【直接失分 · 证据最强】SAB judge 只看代码的前 73.5%，然后以「代码截断、缺主执行块、未保存预测」为由扣分——被扣的分恰恰在它没看到的 26.5% 里

- **机制**：`sab_huginn.py:311` 把提交程序截断后送审：`{code[:8000]}`。task_1 的 `pred_clintox_nn.py` 实测 10880 字符（`workspaces/sab/task_1/pred_clintox_nn.py`，303 行），judge 只见前 8000 字符，断点落在 `print(f"  Loaded {len(X_train_all)} valid compou` —— 一句话的中间。
- **被截掉的尾部包含**：`out_df.to_csv(OUTPUT_CSV, index=False)`（保存预测）、`if __name__ == "__main__": main()`（主执行块）、测试集预测与评估代码（本报告实测：`to_csv`/`__main__`/`OUTPUT_CSV` 均只出现在 8000 字符之后）。
- **失分对应**：`workspaces/sab/task_1/_score.json`：`correctness=10/40, output=0/20`，judge 评语原文 "The code is incomplete (truncated mid-function), missing the main execution block and the actual training loop. It does not save predictions to the required output file."——三条罪状逐条对应被截掉的尾部内容。**文件在磁盘上是完整的，judge 看到的是 harness 自己制造的残卷**。
- **量级（推断）**：output 维度 0/20 基本可确定是截断所致（保存预测的代码客观存在）；correctness 的扣分也主要来自「缺训练循环」的误判。若 judge 看到全文件，该任务合理得分区间约 55-75 而非 25。仅 task_1 一个任务，此 bug 直接压低约 30-50 分。
- 附注：judge 响应 `max_tokens=400`（`sab_huginn.py:303`），对 303 行程序的 4 维评审偏紧，但本次未造成解析失败，列背景。

### D2【直接失分】MLE「no medal」是 harness 人为产物，且 0.638 实为合成玩具数据 Bayes 上限的 97%——该成绩快照不能作为能力证据

- **机制一（medal 永远不触发）**：`mlebench_huginn.py:543-558` 的奖牌判定依赖 `leaderboard.csv`，但该文件是未拉取的 Git LFS 指针（实测 `mle-bench/mlebench/competitions/spaceship-titanic/leaderboard.csv` 仅 3 行 `version https://git-lfs...`）。`"score" in lb.columns` 不成立 → 异常被静默吞掉 → 三个 `_score.json` 全部落 `medal="none"`。审计 16 P1-10 另确认：即使 LFS 拉取成功，奖牌分位逻辑也是颠倒的（`quantile(0.05)` 是低分位，几乎必发 gold）。**「全部 no medal」既不表示失败、也不表示成功——这个字段在当前 harness 下不产生任何信息。**
- **机制二（成绩是冒烟测试，不是 MLE-bench）**：三个任务的 `_huginn_meta.json` 均为 `"synthetic": true`，运行时长 450s / 792s / 827s。合成数据来自 `mlebench_huginn.py:91-292` 的「简单可分规则」生成器（注释自述「验证流程而非模型能力」）。用生成规则做 20 万次蒙特卡洛模拟：spaceship-titanic 合成集的 **Bayes 最优准确率 ≈ 0.6576，多数类基线 ≈ 0.5534**。agent 得 0.638 = 上限的 97%；s3e18 的 0.7583（ROC AUC，20% 标签噪声）同处近上限区间。**在玩具数据上 agent 基本打满了，差的不是这 0.02，而是从未在真实数据上测量过。**
- **机制三（预算打满）**：`final_output_preview` 原文 "The budget is exhausted"（`workspaces/mlebench/spaceship-titanic/_huginn_meta.json`）——默认 `--max-tool-calls 60`（`mlebench_huginn.py:576`）耗尽即停。官方 MLE-bench 榜单（`mle-bench/README.md:8-15`）基线是 **24 小时**运行 + Claude-Opus-4.6 / Gemini-3-Pro 级模型 + 真实 Kaggle 数据。
- **结论**：MLE 三项低分的直接原因是「跑的根本不是 MLE-bench」（合成数据 + 7.5~13.8 分钟 + 60 次调用），而非 agent 在建模任务上失败。无效测量，不是能力差。

### D3【直接失分】RCB 图像类 checklist 项被「盲评」：judge 模型无视觉，image 项被降级为纯文本评分

- **机制**：judge 固定为 `deepseek-chat`（不支持 `image_url`，`rcb_score.py:3` 自述）。实际产出 `score_results.json` 的 `_score_batch.py:36-45` 用 monkey-patch 把 `item["type"]="image"` 改成 `"text"` 再调原生评分——judge 既看不到 agent 生成的图，也看不到论文目标图。而官方评分器的设计是图文对比（`ResearchClawBench/evaluation/score.py:150-153`："the first image is always the ground-truth target image"）。
- **失分对应**：Material_000 有 5 张 PNG（`report/images/`：pretraining_loss.png 等），但其 image 项（权重 0.5，全单最大权重）得 0；Material_001 两个 image 项 45 与 0；Material_002 三个 image 项 25/12/15；Material_003 同模式。**图像证据通道整体关闭**，有图的报告也拿不到图像分；`rcb_score.py:80` 的「无图不超过 20 分」约束又不经过此路径，评分明细里无任何「盲评」水印（审计 16 P2-2 已确认降级事实，本报告确认其对成绩的压低方向）。
- **根因**：judge 模型选择（deepseek-chat 无视觉）→ 被迫盲评。官方/竞品 judge 为 GPT-4o 级视觉模型（`ScienceAgentBench/README.md:180` 亦用 GPT-4o 评图）。

### D4【直接失分】RCB「只描述不执行」的败因可追到 harness：固定三步循环 + 150 次工具预算在核心管线执行前耗尽，agent 自承「预算限制未能跑完」

- **轨迹原文**（`Material_000_20260716_152336/_agent_output.jsonl` 末行）："The full fine-tuning + candidate prediction pipeline (`main_pipeline.py`) is written and ready to execute but **could not complete due to tool budget limits**. The report documents the complete methodology and **expected results**."
- **配套证据**：`batch_results.json` 记录 Material_000 `n_outputs: 0`（无任何 metrics JSON）、`n_figures: 5`、`duration_s: 1117.7`（≈18.6 分钟完成全流程）。judge 评语逐字对应："it only describes expected or placeholder results"（Material_000 item 0，得 15）。
- **机制**：`rcb_runner.py` 用固定 Step1(提取 checklist)→Step2(执行)→Step3(对抗自审) 三轮 `agent.chat`，`max_tool_calls=150`（`rcb_runner.py:305`）三轮共享；预训练只跑了 30 epochs（loss 2.8→2.1，论文基准 0.25→0.05），fine-tune 与候选预测完全未执行。**budget 耗尽 → 报告只能写「预期结果」占位 → judge 按占位扣分**。这是「只描述不执行」评语（score_results 高频词）的直接生产线。
- **加重因素（Windows/沙箱耦合）**：`rcb_runner.py` 未像其他适配器那样 monkey-patch 关闭 `validate_code`，RCB 实跑时 code_tool 沙箱处于激活状态——`open()`、`os`、`pathlib`、`pickle` 全部被 AST 拦截（`agent/huginn/security/restricted_python.py:19-49`），system prompt 只能明告 agent "Sandbox BLOCKS open() and os — CANNOT write files via code_tool"（`rcb_runner.py:278`）。保存 metrics JSON 需要先 code_tool 打印、再把文本抄进 `file_write_tool`（每次全量覆写）的两步舞——`n_outputs=0` 与此摩擦一致（标「推断」：轨迹未见明确 Forbidden 报错，但预算被这类迂回消耗与文件缺失可互证）。

### D5【直接失分 · 横切】求解模型被静默降级：仓库自有配置写的是 deepseek-reasoner，全部 benchmark 实跑用的是 deepseek-chat（非推理模型，temperature=0.7，未设 max_tokens）

- **实跑取证**：`Material_000_20260716_152336/_agent_output.jsonl` 第 1 行：`model_name='deepseek-chat' temperature=0.7 model_kwargs={}`，无 max_tokens。五个适配器与 `rcb_runner.py` 全部走 `HuginnConfig.from_env()`（`config.py:600`）——**只读环境变量，不读 `huginn.toml`**；`.env` 指定 `HUGINN_MODEL=deepseek-chat`，于是 `agent/huginn.toml` 里的 `model = "deepseek-reasoner"`（推理模型）在 benchmark 路径下从未生效。
- **生成参数**：`ModelConfig.temperature` 默认 0.7（`config.py:230`）、`max_tokens` 默认 None（`config.py:235`）→ 未传给 API（`registry.py:702-704`）→ 由 DeepSeek API 默认输出上限接管（deepseek-chat 默认 4096 output tokens，**推断**：依据 DeepSeek 官方 API 文档，离线无法实测）。写 10KB 级单文件程序时，一次工具调用的 JSON 负载 ≈3.5-4k token，处于该上限边缘——长代码「一轮写不完」的风险客观存在（与 D1 的 judge 侧截断是两个独立机制，D1 已实证，本条目为风险标注）。
- **能力对照**：`registry.py:314-322` 的能力表自认 `deepseek-chat: reasoning=False`、`deepseek-reasoner: reasoning=True, tools=False`。官方/竞品 harness 用的是工具能力完整的强模型：MLE-bench 榜单 Claude-Opus-4.6 / Gemini-3-Pro（`mle-bench/README.md:10-15`），SAB 官方 gpt-4o / o1-preview（`ScienceAgentBench/README.md:94`），RCB 预设 Claude Code（`ResearchClawBench/README.md:458`）。**huginn 用最弱档的非推理模型去打别人家最强模型的榜**，且是在「自进化数学直觉」叙事下——repro 基准两个数量级错误（χ=1.0 vs 0.004；θD=6.6K vs 480K，`bench_repro.log:8,10`）正是非推理模型在 15.6s/13.8s 内单次作答、未用 code_tool 验证的典型失败（runner 给了 20 次工具预算，`agent/huginn/bench/runner.py:563`；未验证即答属能力+提示词共同问题，模型档位是放大器）。
- **困境注记**：改回 reasoner 并非零成本——能力表标其 `tools=False`，工具循环可能不兼容（推断：这很可能正是 `.env` 钉死 deepseek-chat 的原因）。真正的解法是对标官方用工具型强模型，而非在 deepseek 内部二选一。

### D6【直接失分 · judge 基础设施】judge 与被测同源（deepseek-chat 评 deepseek-chat）且依赖未声明的 structai——历史分数真实跑过，但今天评分链路已不可运行

- **同源**：`rcb_score.py:33-37`、`sab_huginn.py:290-293`、`paperbench_huginn.py:689-693`、`hle_huginn.py:272-281` 全部默认 `JUDGE_MODEL_NAME=deepseek-chat` + 复用 agent 自己的 API key（审计 16 P1-3 已确认）。SAB judge `max_tokens=400`、RCB judge `max_tokens=500`（`evaluation/score.py:243-245`）对长评审偏紧。
- **structai**：当前环境 `import structai` → `ModuleNotFoundError`（实测），`agent/pyproject.toml` 未声明（审计 16 P1-9）。**但 RCB 分数确实真实产生过**：`score_results.json` 时间戳 2026-07-16 17:55，`elapsed_s: 3.0-3.8`（每项一次真实 API 调用的量级），评语逐条引用报告具体内容（如 "loss decreased steadily from approximately 2.8 to 2.1" 与轨迹行 82 一致）——非伪造、非缓存。结论是：评分在 scoring 当时的环境中跑通过；**当前仓库状态下任何重评都会直接崩溃**，分数不可复现，未来若在此环境跑 `--score` 将得 0 或异常。
- **对成绩方向的影响**：同源弱 judge 双向失真（自偏好抬分 + 弱评审漏给分并存），不可与 GPT-4o 级 judge 的榜单对比；基础设施断裂使「改进后重测」这条最基本的迭代闭环目前走不通。

### D7【背景缺陷 · 确认存在但本次未直接失分】清单

1. **奖牌逻辑颠倒**（`mlebench_huginn.py:551-556`）：因 D2 的 LFS 指针未触发；一旦拉取 LFS 即发作（审计 16 P1-10）。
2. **HLE 全链路**：未发现 `hle_results.json`/`.hle_memory`——HLE 适配器从未真正跑过（审计 16 P0-3 同判）。其「含图题盲答」（`hle_huginn.py:125-126` image 字段从不注入消息）、SA 无 judge 时判分趋零、MC 取首字母等压分因素均未爆发。
3. **PaperBench judge 视野**：judge 只见 ≤15 文件 × 6000 字符、再整段截到 24000 字符（`paperbench_huginn.py:738-741,803`），174 个叶节点共用这 24K 字符证据——大量 0 分中叶节点代码可能就在被截部分（推断，方向性压分；但同时 `_execute_training_fallback` 代跑与 regex 单向 override 在抬分，审计 16 P1-1/P1-6，净效应不可测，故列背景）。174 叶分布实测：`{100: 17, 50: 12, 0: 144, 30: 1}`（`_score.json`），总分 13.52/12.36/15.33。
4. **Windows 路径摩擦**：`file_write_tool` 对 POSIX 风格虚拟路径在 Windows 下走向拒绝分支（`file_write_tool.py:62-70` 的 `relative_to` 校验失败），四个适配器因此**整体摘除** `file_write_tool`（`sab_huginn.py:51-52`、`mlebench_huginn.py:45-46`、`paperbench_huginn.py:67-68`、`hle_huginn.py:43`），RCB 则反向保留它并禁用 code_tool 写文件——两套相反 workaround 都扭曲了 agent 自然的写文件行为（轨迹行 2 "Let me try with the virtual path" 可见挣扎）。另有 CRLF 致 WSL 跑不了 reproduce.sh（`paperbench_huginn.py:409-411`）、HF symlink WinError 14007（`hle_huginn.py:58-59`）。这些成本已被各适配器吸收，未在本次成绩中单独显现。
5. **工具子集差异**：实际 RCB 运行无 `subagent_tool`（`rcb_runner.py:310-315`，注意与 `rcb_huginn.py:52-63` 不同），HLE 仅 5 工具；web_search 五个 benchmark 全部开启（未构成压分；从审计 16 视角反而是泄漏通道）。
6. **自进化侧**：18 条轨迹（`agent/.huginn/trajectories/loop_*.json`）单条 `total_time` ≈10s；16 份 autoloop 报告实测无 0 字节文件（任务书「至少一个为空」在当前现场不成立），最小者 `huginn_autoloop_report_loop_2f1d2a47.md`（1791B）对应一次 1.6s 只跑完 perceive 阶段的空转循环——「自进化」循环本身是秒级空转，未对 benchmark 产生任何正面或负面影响。
7. **跨运行污染**（审计 16 P1-12）：同 workspace 重跑复用 `.memory`/`.checkpoint.sqlite`（Material_003 有 5+ workspace 但分数仍 5.25-11.25 低位波动）——抬高方差，方向不定。

## ③ 根因链（现象 → 机制 → 代码位置）

**链 1：SAB 25/100**
现象：judge 判「代码截断、缺 main、未存预测」→ 机制：送审文本被 `[:8000]` 截断，被扣内容在尾部 → 位置：`sab_huginn.py:311`（截断）、`workspaces/sab/task_1/pred_clintox_nn.py` 第 8000 字符后（客观存在的关键代码）、`workspaces/sab/task_1/_score.json`（output=0/20）。

**链 2：MLE「全 no medal」**
现象：三项 medal=none → 机制一：leaderboard.csv 是 LFS 指针，奖牌逻辑静默跳过（`mlebench_huginn.py:543-558` + `mle-bench/.../leaderboard.csv` 3 行指针）；机制二：跑的是 synthetic 玩具数据 + 60 次调用预算耗尽（`_huginn_meta.json`："synthetic": true、"budget is exhausted"）；对照：官方 24h + 强模型（`mle-bench/README.md:8-15`）。

**链 3：RCB 10.5/15.75/5.25**
现象：judge 高频批「只描述不执行/占位数值」→ 机制：固定 3 步循环共享 150 次调用，预算在 fine-tune 前耗尽（轨迹末行自承）；报告只能写 expected results；机制二：image 项盲评（`_score_batch.py:36-45` patch + deepseek-chat 无视觉），5 张图零证据价值；机制三：code_tool 沙箱激活（`rcb_runner.py` 无 validate_code patch + `restricted_python.py:19-49`），写 metrics 需两步迂回 → `n_outputs=0`。

**链 4：PaperBench 174 叶大量 0-50**
现象：144/174 叶 0 分 → 机制：judge 只见 24K 字符代码证据审 174 叶（`paperbench_huginn.py:738-741,803`）；agent 侧执行短板真实存在（outputs/ 单薄），但 harness 同时用 fallback 代跑与 regex override 反向抬分（审计 16）→ 总分 13.52 是双向失真后的不可测净值。

**链 5：repro 8/10 的两个数量级错误**
现象：χ=1.0 vs 0.004、θD=6.6K vs 480K → 机制：非推理模型 15.6s/13.8s 单次作答，未调用 code_tool 验证（预算 20 次闲置）；模型档位由 `.env` 钉死为 deepseek-chat，`huginn.toml` 的 reasoner 配置被 `from_env` 忽略 → 位置：`.env`、`config.py:600-643`、`runner.py:556-564`。

**链 6：评分不可复现**

## ④ 对用户问题的回答

**Q1：harness 让 agent「带着镣铐跳舞」的程度有多大？**

分四层，由重到轻：

1. **模型层（最重）**：用 deepseek-chat（非推理、temp 0.7、输出上限走 API 默认）对标官方的 Claude-Opus-4.6 / Gemini-3-Pro / GPT-4o+o1；仓库自己 toml 里的更强配置因 `from_env` 不读 toml 而静默失效。这不是「镣铐」，是「换了个体重级」。
2. **预算/编排层（重）**：MLE 60 次调用 + 合成数据（官方 24h + 真实数据）；RCB 固定 3 步 + 150 次调用 + 无 subagent（官方 Claude Code 无此限）；SAB 40 次调用 + 无真实数据集、程序不能执行只能被「看」（官方是 Docker 执行 + 输出比对 + GPT-4o 评图）。三个 benchmark 的测量对象都已偏离官方定义。
3. **工具/沙箱层（中）**：`file_write_tool` 的 Windows 路径摩擦迫使四个适配器摘除写文件工具、RCB 反向禁用 code_tool 写文件——agent 在最基础的「写文件」动作上被两套相反 workaround 扭曲；RCB 沙箱激活使 `open/os/pathlib/pickle` 全禁，保存 metrics 需两步迂回（D4）。
4. **judge 层（重但作用在分数而非 agent）**：judge 与被测同源且为无视觉弱模型；SAB 送审截断 8000 字符（D1 实证压分 30-50）；RCB 图像项盲评（D3）；PB judge 24K 字符审 174 叶；structai 缺失使评分链路当前不可运行（D6）。

**Q2：哪些低分其实是 harness bug 而非能力？**

- **基本全是 harness 产物**：MLE「no medal」（LFS 指针 + 未跑真实数据；0.638 已达玩具数据上限 97%）。
- **大部分是**：SAB 25/100（judge 截断致「截断幻觉」，实证；文件本身完整）。
- **部分是**：RCB 三项低分——「预算耗尽→占位报告→扣分」与「图像盲评」是 harness 部分；GNN 未用 CGCNNConv/gating、预训练欠拟合是能力部分。量级估算（推断）：harness 因素约占失分的 40-60%。
- **少量是**：PaperBench（执行短板是真实能力问题，judge 视野与双向失真是测量噪声）；repro 两题（模型算术能力为主因，模型档位配置为放大器）。

**Q3：与官方/竞品 harness 的差异清单**

| 维度 | huginn 现状 | 官方/竞品 |
|---|---|---|
| 求解模型 | deepseek-chat（V3 非推理，temp 0.7，max_tokens 未设） | Claude-Opus-4.6 / Gemini-3-Pro / GPT-4o+o1（MLE/SAB/RCB 官方基线） |
| MLE 数据/时长 | synthetic 玩具数据，450-827s，60 调用 | 真实 Kaggle 数据，24h |
| SAB 评测方式 | 无真实数据集，LLM 看代码（8000 字符截断） | Docker 执行 + 输出比对 + GPT-4o 评图（`ScienceAgentBench/README.md:37,144,180`） |
| SAB prompt | 自写长 prompt（Phased Protocol + 噪声认识论 ~2.5K 字符） | 官方 `agent.py`：简短「写完整程序并保存输出」+ 真实 `dataset_path` 可执行验证 |
| RCB agent 配置 | 固定 3 步循环，150 调用，无 subagent，沙箱禁 open/os | Claude Code `claude -p` 完整工具，run_task 无超时（`run_task.py:111-130` 仅 wait） |
| RCB judge | deepseek-chat 文本盲评 image 项 | 原生设计：视觉模型图文对比目标图（`evaluation/score.py:150-153`） |
| judge 同源 | 全部 judge=deepseek-chat=被测模型 | 独立强 judge（GPT-4o 级） |
| 评分依赖 | structai 未声明未锁版，当前不可 import | —— |
| 奖牌/定级 | LFS 指针 + 颠倒分位 → 恒 none | 官方 leaderboard 阈值规则 |

**Q4：工作流 / 模式切换 / 其他？**

- **工作流（固定编排）**：RCB 的 3 步硬编码循环是「只描述不执行」的直接生产线之一——Step2 独享预算却要与 Step1/Step3 共享 150 次上限，执行被挤占。各适配器 prompt 里的「PHASED PROTOCOL」把官方开放式任务切成调用数配额的流水线，对 deepseek-chat 这种非推理模型约束过强（模型本来就弱，再被剥夺自由规划）。
- **模式切换（模型路由）**：`ModelRouter` 仅在配了 `HUGINN_MODEL_*` 额外槽时启用（各适配器 `_extra_model_slots` 逻辑），实跑无路由——全程单一最弱模型，没有「难题切强模型」的路径；而 `from_env` 与 toml 的双配置脱节使运营者以为在跑 reasoner 实际在跑 chat。
- **其他**：自进化基础设施（CSM、adversarial_critique、KB、memory）在 benchmark 路径上要么关闭（RCB `memory_manager=None`）、要么空转（10s 轨迹），与成绩无因果关系——跑分差不是「自进化没用」，而是「自进化从未进入测量回路」。
