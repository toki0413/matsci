## ② 核心发现（按证据强度排序）

### F1 【直接证据 · 最强】autoloop 是「 perceive 空转 + report 兜底」，18/18 轨迹零工具调用

对 18 个轨迹逐一解析（脚本统计，非抽查）：

- **phase 构成**：66 × `perceive` + 18 × `report`，**0 × hypothesize / plan / execute / validate / learn**。单轨迹最长 11.1s，最短 1.5s。
- **`tool_calls` 全部为 `[]`**——18 个轨迹合计 0 次工具调用（每个 JSON 的 `tool_calls` 字段，如 `agent/.huginn/trajectories/loop_36b59bd5.json:43`）。
- 全部 span `status=completed`，包括 3 个 objective 为 "Compute something that will fail" 的运行（`loop_7db8c43b/a362d257/fe3d6d8e`）——**连故意的失败目标都显示「完成」**。

机制定位：`agent/huginn/autoloop/engine.py`

- `_perceive()` 在 `not snapshot.has_activity()` 时返回 `None`（:2304-2305；legacy 路径 :2388-2389 同样「无 git 变更且无错误日志 → None」）。
- `run()` 收到 `None` 后打日志 "no changes detected, waiting..." 并 `continue`（:1353-1362）——**跳过 hypothesize→learn 全部六个阶段**。
- 循环结束后 `_finalize_run` **无条件**生成报告（:2160-2163），`success = all(p.status == "completed" for p in phases[-7:])`（:2269）——perceive/report 永远 completed，故 success 恒真。

**结论：所谓「自主研究循环」是文件监听事件驱动的看门狗，不是 objective 驱动的研究器。** 给它一个研究目标，它等文件变化；文件不变，它空转 N 轮后写一份报告。

### F6 【背景缺陷确认】PhaseGateState 单例串扰：本批运行中未起效，但使「修好之后」的门控仍不可信

引用 audit 04 P1-4 已确认事实（`audit_20260717/04_架构_核心智能体循环.md:122-148`）：

- `phase_gate.py:692-702` 模块级单例；`engine.py:1265` 每个 `run()` 开头 `reset_runtime()`；`routes/autoloop.py:91-94` 每 HTTP 请求 `create_task` 一个新 run，**无并发互斥**。
- 影响：并发 run 互相清 gate history/pending review/overrides；`_wait_if_checkpoint_pending` 轮询共享状态，硬门等待会「互相放行或互相卡死」；交互式 `phase_tool` 可污染后台 run。

**对本次诊断的归因定性**：在已勘察的 34 次运行中，循环从未推进到 plan→execute 门（F1），因此单例串扰**不是本批空转的直接原因，属背景缺陷**。但它的存在意味着：即使修复 F1 让循环真正跑起来，质量门（red-team 审查、人工 checkpoint）在并发或人机混用下仍会误判放行——gate feedback 会经 `_speculator_hint` 注入**错误 run** 的下轮 prompt（engine.py:857-865），二次污染循环有效性。修复优先级应排在 F1 之后、任何「依赖门控保证质量」的宣称之前。

### F7 【直接证据】测试断言空洞，使 F1–F3 在 CI 中不可见

`agent/tests/stress/test_sci_automation.py:269-292`：名为 `test_autoloop_completes_all_phases` 的用例，断言仅 `assert result is not None` + `hasattr(result, "phases")`；`test_autoloop_handles_tool_failure` 仅 `assert result is not None`。这解释了为什么「7 阶段循环实际只跑 2 阶段」能长期存活——**质量闸门本身没有牙齿**。

## ④ 对用户问题的回答

**Q：所谓自进化循环是真「进化」还是「空转」？**
空转。判定依据：18/18 轨迹 perceive+report 之外无任何阶段、0 工具调用（F1）；25 条进化规则 usage_count 全 0（F3）；同一已知错误（虚拟路径）在经验成文数月后的最新运行中原样复发 25 次（F4）；同一 objective 跨 2 天重跑产出逐字节同构（F5）。策略/代码/提示词均未被循环自身修改并留存——唯一的 prompt 变更来自人工（F5 机制 5）。

**Q：trajectory 之间是否存在可观测的能力改进趋势？**
不存在。34 次运行两批，phase 构成只有两种模板（perceive×1+report / perceive×5+report），耗时 1.5–11.1s，全部 completed，无失败记录、无策略差异、无指标可追踪。trajectory 文件本身也不记录 hypothesis/plan 内容，即使想回溯趋势也无数据。

**Q：PhaseGateState 单例串扰对循环有效性有何影响？**
本批运行中**无直接影响**（循环从未到达任何门控点，F6），属背景缺陷。潜在影响有二：(a) 修复空转后，并发 run 会互相 `reset_runtime()` 清状态，硬门等待互相放行/卡死（audit 04 P1-4）；(b) 被阻断 run 的 feedback 会经 `_speculator_hint` 串入另一 run 的 prompt，造成跨 run 证据污染。结论：**它当前不是失分原因，但它是「循环有效性」的未爆弹**——任何基于门控的质量保证宣称在修复前都不可信。

**Q：失分主因是工作流、模式切换还是其他？**
是「其他」中的两项，且都不是模式切换问题：(1) **执行缺位**——SAB output=0、PaperBench Code Execution 0%、RCB「只描述不执行」，跨 benchmark 同病理；(2) **交付物纪律缺失**——不保存预测文件、不写 reproduce.sh。模式切换/工作流编排（39 号文的 4 层 Orchestrator）是次要因素。自进化子系统对上述两项零干预。
