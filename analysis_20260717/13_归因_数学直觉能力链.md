## ② 核心发现（按证据强度排序）

### F1 ｜ 五个外部 benchmark 全部用白名单把数学工具整体摘除 —— 数学能力在跑分中不是"没被调用"，是"根本不可用"（证据：代码 + 轨迹双重确认，强度最高）

四个外部适配器均显式定义 8–10 个通用工具的 filter，**无一包含** `symbolic_math_tool` / `bourbaki_tool` / `lean_tool` / `unit_tool` / `validate_tool` / `numerical_tool`：

- `sab_huginn.py:53-62`：`SAB_TOOL_FILTER = ["code_tool","bash_tool","file_read_tool","glob","grep","web_search_tool","subagent_tool","plot_tool"]`
- `rcb_huginn.py:52-63`：`RCB_TOOL_FILTER` 同构（注释："其他 80+ 工具只会分散注意力 + 占 context tokens"）
- `mlebench_huginn.py:47-57`、`paperbench_huginn.py:71-79`：同构

轨迹侧交叉确认：`ResearchClawBench/workspaces` 全目录 grep `sympy|symbolic_math_tool|bourbaki|lean_tool|unit_tool` **0 命中**；最大的单次轨迹（`Material_003_20260716_190227/_agent_output.jsonl`，1.03 MB）中 `'sympy':0, 'symbolic':0, 'units':0, 'order of magnitude':0`；SAB/MLE/RCB 各 workspace 残留的 `_code_tool_script.py`（最后一次 code_tool 脚本）sympy import 均为 0。

**结论**：在全部外部 benchmark 中，huginn 的符号推导、量纲检查、守恒验证、形式化证明、数值-符号互验五条链**结构性缺席**。这不是 agent "没想到用"，是 harness 配置层先验地摘除了这些器官。

### F2 ｜ 内部 repro 基准中数学工具其实可用，agent 仍然裸答出错 —— 第二层断点在行为，不在工具（证据：runner 代码 + 日志）

内部 bench（`agent/huginn/bench/runner.py`）走的是全量注册：`register_all_tools()`（:392）+ `agent.register_tools_from_registry()`（:566），system prompt 还内联了 `MATH_DEPTH_GUIDE`（:553-554，该指南见 `prompts.py:373-411`，明确宣传 `symbolic_math_tool` 的 pde_classify/euler_lagrange/noether/diffgeo 等 action）。即便如此，`bench_repro.log` 记录：

```
[6/10] repro-curie-weiss ... FAIL (15.6s) χ=1.0, 期望 ~0.004
[8/10] repro-debye-temp ... FAIL (13.8s) θD=6.6 K, 期望 ~480 K
```

两题都是**一次计算器调用即可解的纯算术**（χ = C/(T−θ) = 1/(300−50) = 0.004；θD = hνmax/kB = 6.626e-34×1e13/1.381e-23 = 479.6 K）。错误形态：χ=1.0 恰等于题面给的 Curie 常数 C（照抄输入）；θD=6.6 K 的尾数与 h 的尾数 6.626 一致（推断：指数处理丢失约 72 倍）。失败耗时 13.8–15.6s，为 10 题中最短的两档（PASS 题如 arrhenius 用了 65.9s）——符合"未走任何计算工具、直接文本作答"的时间特征（推断，bench 会话日志落盘于 `~/.huginn/logs/`，在本次只读范围内未取证）。

**结论**：即使工具可用、prompt 宣传到位，agent 也没有"数值必须算出来"的行为约束。bench system prompt 写的是建议（"Use code_tool to compute when arithmetic is non-trivial"，`runner.py:544`），不是 gate。

### F3 ｜ 形式化验证链的"验证"内容是重言式；Lean 验证完整性已被审计证伪（证据：代码 + 审计 07）

- `tools/bourbaki_tool.py:116-132`：Lean 守恒检查的 `lean_code` 是**硬编码的平凡实例**——`myEvolution (s : ℝ) : ℝ := s`（恒等演化）、`invariant := id`（恒等不变量），`preserved := by intro s; simp`。无论用户传入什么方程，证明的都是"恒等映射保持恒等不变量"这一重言式，与用户方程无关。
- `lean/sympy_to_lean.py:164`：定理骨架生成器固定输出 `:= by sorry`。
- 审计 07 P2-3（`audit_20260717/07_工程_工具与技能层.md:200-214`）已确认：`LeanInterface.verify_theorem`（`lean/interface.py:158-191`）只做"模块编译通过 + 定理名子串匹配"，`by sorry` 占位证明、注释中出现的定理名均可通过"验证"；全库无 `sorry`/`axiom` 扫描。
- `bourbaki_tool.py:186-202`：无 Lean 时的 fallback 多条路径返回**罐头文本**（如 `buckingham_pi` 恒返回 "3 dimensionless pi_groups [Re, Fr, We]"，`dimensional_analysis` 只检查单位名是否在 5 个基本量名单内，:212-244）。

正面事实：Lean 工具链在本机就绪（`agent/lean/HuginnLean/lean-toolchain` = `leanprover/lean4:v4.16.0`，`~/.elan/bin/lake.exe` 存在），HuginnLean 项目有 11 个模块（BornStability/Elasticity/ContinuumMechanics/DFT/Thermodynamics 等）和 `.lake/build` 构建痕迹；`lean/auto_pipeline.py`（1020 行）实现了 SymPy→Lean Float 表达式编译 + 编译错误规则修复反射环（:291-310）+ 有限差分 vs 符号导数互验（`verify_derivative`，:402-498）。项目目录残留的 `tmp*.lean` 文件是该 pipeline 真实运行过的痕迹（来自测试，非 benchmark）。

**结论**：形式化证明子能力处于"工具链与 AST 桥就绪、验证语义为空"的状态——它验证的是"表达式能编译成 Lean Float 代码"，不是"物理命题为真"。

### F4 ｜ 约束/守恒系统接了线，但只挂在仿真工具上，且为 warn 级 —— 与 benchmark 失分面零交集（证据：代码）

- 接线点：`tools/adapter.py:188`（类级 `ConstraintAdapter.default()`）、:425-437（仅当 `result.success` 且工具声明了 `constraint_scope` 才评估；失败结果直接放行）。
- `constraint_scope` 只声明在仿真类工具上（`fem/tool.py:73`、`sim/vasp|lammps|qe|cp2k|gromacs|openfoam|...`、`mechanical_tool.py:215`）——benchmark 中 agent 自己写的 pandas/sklearn 代码（code_tool）**不在任何 scope 内**。
- 默认约束库（`constraints/adapter.py:292-324`）：全局仅 `finite_values`（block）；质量项为 dft（能量符号/力收敛/带隙/体积/磁矩）、md（能量守恒/温度稳定/原子数/密度）、phonon（虚频）——全是仿真结果 sanity 检查，没有"数值答案量级是否合理"的通用检查。审计 14 另发现其中两条判据本身不当（温度涨落阈值无 1/√N 标度、虚频对过渡态必误 block，P2-7）；`execution/physics_auditor.py` 还有压力单位换算错误（100000 bar 实为 10 GPa 被当 100 GPa，审计 14 P2-3）。

### F5 ｜ 「自进化」loop 的 18 条轨迹零工具调用、只做 perceive+report —— 自我进化机制从未触达数学验证层（证据：轨迹 + 报告原文）

脚本统计 18 个 `agent/.huginn/trajectories/loop_*.json`：`tool_calls` 字段**全部为 0**，phases 仅 `perceive`×N + `report`。对应报告佐证：`agent/huginn_autoloop_report_loop_2f1d2a47.md`（16 份报告中最小，1791 字节；16 份均非空）记录总时长 **1.6s**、仅 perceive 完成，其 Tutor's Summary 自承："the loop only ran *perceive*. The *act* phase (applying strain and recalculating) didn't execute—perhaps the loop was cut short."

autoloop 引擎内确实有数学验证钩子（`autoloop/engine.py:4131-4209`：`BourbakiTool.check_conservation` + `LeanTool.constitutive/variational_principle` + `AutoDiffTool.gradient`），但它要求上游 `execution_result` 携带结构化 `"equations"`/`"lagrangian"` 字段才触发——benchmark 适配器全部绕过 autoloop 直接 `HuginnAgent.chat()`，该钩子与跑分路径零交集；18 条轨迹表明 autoloop 自身运行也未走到执行/验证相。

`cognitive_primitives.py` 全文 1 行：`# deprecated: adversarial_critique 直接塞 rcb_runner.py 了`（审计 04 P3-1 已标记为死文件）。`metacog/method_registry.py:60` 的 `bourbaki-structure` 方法族自标 `(advisory)`，唯一消费方是 autoloop（`engine.py:2485-2487`）。`ml/transfer_registry.py` 仅被 `academic/deli_research.py:1449` 与 `autoloop/conjecture.py:721` 引用。`execution/dimensional_validator.py`（完整的代数量纲系统 + Buckingham π + SymPy 维度推断）仅被 `ontology/actions.py:255,340` 以字符串名引用 + 测试。**这些模块互不相干地活在 autoloop/ontology 孤岛里，与五个 benchmark 执行路径均不相交。**

### F6 ｜ RCB 的直接失分机制是"执行中断→占位写作"，不是"缺数学工具"（证据：轨迹原文 + judge 评语）

## ③ 根因链（现象 → 机制 → 代码位置）

### 链 A：repro 基准 χ=1.0 / θD=6.6K（数量级错误）

**现象**：`bench_repro.log` 2/10 FAIL，错误值为"照抄题面常数 / 尾数对而指数丢"形态。

**机制**：agent 以纯文本推理直接提交数值答案，全程无计算工具调用；系统无任何"数值结论必须有计算来源"的 gate；判分器 `_eval_num_in_output` 提取输出中全部数字任一命中即过（审计 16 P1-7），反向也无"未计算即答"的惩罚。

**代码位置**：`agent/huginn/bench/runner.py:484-486`（prompt → chat → 直接取 final 文本）；:537-554（system prompt 仅有建议性 "Use code_tool to compute"，无强制）；`paper_repro_bench.py:122-135,153-156`（regex 提数判分）。

**断点定性**：行为层断点（工具可用而不用），非工具层断点。

### 链 B：RCB「只描述不执行 / 占位数值」（10.5 / 15.75 / 5.25）

**现象**：judge 高频评语"only describes expected or placeholder results"、"admits omissions but does not attempt"。

**机制**：(1) 执行基础设施不稳定——bash_tool 跑 sklearn 持续崩溃、code_tool 在 GP+RDKit+sklearn 组合下失败、loop detector 触发（轨迹原文，F6）；(2) 预算/时间耗尽后 agent 选择"写预期值"而非"承认未完成"；(3) Step 3 sanity check 是纯 LLM 文本对照，机制上识别不了"数值无产物支撑"；(4) 数学/验证工具在 filter 外，即使想验证也无可调用者。

**代码位置**：`rcb_huginn.py:52-63`（工具白名单）；`rcb_runner.py:305-306`（`max_tool_calls=150`）；`rcb_runner.py:493-499`（文本式 sanity check）；轨迹 `Material_003_20260716_190227/_agent_output.jsonl`。

**断点定性**：执行稳定性 + 交付完整性断点为主，验证工具缺失为背景缺陷（在本任务中未起作用——agent 连主流程都没跑完，轮不到验证）。

### 链 C：SAB 25/100（output=0）

**现象**：breakdown `correctness:10, output:0, quality:10, science:5`（`workspaces/sab/task_1/_score.json`）。

**机制**：交付物（预测 CSV、主执行块）位于文件尾部，judge 只见前 8000 字符 → output 维度 0 分；部分为评测 artifact。

**代码位置**：`sab_huginn.py:311`（`code[:8000]`）；审计 16 P1-11。

**断点定性**：评测设计 artifact + 交付完整性，与数学能力链无关。

### 链 D：MLE-bench no medal（0.758/0.638/0.745）

**现象**：三任务 medal=none（`workspaces/mlebench/*/_score.json`）。

**机制**：ML pipeline 执行力问题（特征工程/调参深度不足，`_huginn_meta.json` 显示 budget exhausted）；奖牌判定逻辑颠倒 bug 未爆但 leaderboard LFS 未拉取致恒 none（审计 16 P1-10）。

**断点定性**：ML 工程执行力，数学工具无可归因作用面。

### 链 E：PaperBench 大量 0–50 分叶节点

**机制**：长任务执行未完成 + harness 代跑训练兜底失真 + judge 抬分指令（审计 16 P1-1/P1-6 已定论）。C2ST/MCMC 等数值正确性验证本可由数值-符号互验链承担，但 agent 连可运行的 submission 都未稳定产出，验证链无用武之地。

**断点定性**：执行链断裂为主，验证链缺失为背景。

### 链 F（总根因）：「数学直觉」组件的孤岛化

**现象**：组件库丰富（symbolic_math 30+ actions、auto_pipeline 1020 行、dimensional_validator 全代数量纲系统、constraints 约束库、HuginnLean 11 模块），benchmark 全输。

**机制**：三层断裂叠加——
1. **配置层**：外部 benchmark 适配器用 tool_filter 把数学工具整体摘除（链 A–E 的共同前提，F1）；
2. **行为层**：即使工具可用（内部 bench），无"数值必须经计算/验证"的强制 gate，agent 默认裸答（链 A，F2）；
3. **语义层**：被接线的验证中，Lean 守恒检查是重言式、定理验证可被 sorry 通过、fallback 返回罐头文本（F3）——"验证通过"不携带信息量。

## ④ 对专项问题的回答

### Q1：五类子能力各有几成？

按「实现完成度 / 主循环接入度」两维打分（10 成制，证据见 ②③）：

| 子能力 | 实现 | 接入 | 依据 |
|---|---|---|---|
| 符号推导（symbolic_math：derive/solve/pde/euler_lagrange/noether/diffgeo/tensor，30+ actions） | 6 | 2 | 实现侧：8 子模块 + 测试齐全（`tools/symbolic_math/tool.py:18-31`）；接入侧：默认 agent 注册 + `MATH_DEPTH_GUIDE` 宣传，但五个外部 benchmark filter 全部排除（F1），唯一深度消费方是 autoloop 孤岛（F5） |
| 量纲检查 | 5 | 1 | 实现侧两套并存且深浅不一：`execution/dimensional_validator.py`（代数量纲 + Buckingham π + SymPy 推断，深）只接 ontology 字符串引用；`bourbaki_tool`/`symbolic_math` 路径浅（只认 5 个基本量单位名，`bourbaki_tool.py:212-244`）。无一套在 agent 主循环 |

| 形式化证明（Lean4） | 2 | 1 | 工具链与项目就绪（v4.16.0，11 模块），AST 桥（sympy_to_lean）与 auto_pipeline（1020 行）存在；但 verify=子串匹配、sorry 可通过、生成定理自带 sorry（F3）——是"Float 表达式编译器"而非证明系统；接入仅 lean_tool（filter 外）+ autoloop 钩子（未触达） |

### Q2：这些能力在跑分任务中被实际调用了吗？

**没有。三重证据**：
1. 配置证据：四个外部适配器 tool_filter 不含任何数学工具（F1，逐行列出）；
2. 轨迹证据：30 份 RCB 轨迹全目录 grep 0 命中；18 份自进化轨迹 `tool_calls` 为 0；SAB/MLE/RCB 残留脚本 sympy import 为 0（F1/F5）；
3. 行为证据：内部 bench 工具可用仍裸答（F2，repro 两 FAIL 题耗时为全 10 题最短档）。

### Q3：接入它们对哪类失分本可起多大作用？

| 失分类别 | 反事实收益 | 机制与上限 |
|---|---|---|
| repro 数量级错误（χ=1.0、θD=6.6K） | **高且直接**——2/10 题可直接翻转，pass rate 80%→100% | 但杠杆不是"更多数学工具"（内部 bench 已全量可用），而是**强制计算 gate**：数值答案必须携带计算工具调用记录。一行 sympy/code_tool 调用即可防 |
| RCB「占位数值」（10.5–15.75 分区间的主要压分项） | **中**——且首解不是数学工具 | 根因是执行中断（基础设施崩溃 + 预算耗尽）。对症的是 (a) 修复 bash/code_tool 稳定性，(b) 确定性的"报告数值声明 vs outputs/ 产物"一致性核对器。量纲/守恒检查对"MD 验证候选 Tg"这类声明有辅助价值，但 agent 未跑到那一步 |
| SAB 25 分 | **近零**——失分主因是 judge 视野截断 artifact + 交付完整性（F7） | 修 `code[:8000]` 与 prompt-评分细则循环（审计 16 P1-11）即可回收大部分 |
| MLE-bench no medal | **近零** | ML pipeline 执行力问题 |
| PaperBench 叶节点 0–50 | **低-中（边际）** | 若 C2ST/MCMC 实现能跑通，数值互验可防"算错而不自知"；当前主断点在执行链 |

**一句话**：数学能力链对当前成绩单的可归因失分**上限约 1.5 个 benchmark 小项**（repro 2 题 + RCB 部分定量项）；它是"第二阶修复"，第一阶是执行稳定性与交付完整性。

### Q4：最短闭环建议

**repro 类失分的最短闭环**（~50 行代码，无需碰 Lean/sympy 生态）：
在 `bench/runner.py` 的 `_run_task` 判定前加确定性 gate——若任务期望数值答案（evaluator 含 `_extract_number`）而 agent 全程 `code_tool`/`symbolic_math_tool` 调用数为 0，则 (a) 打回一次并注入 "You must compute, not guess" 指令，(b) 仍裸答则判 FAIL。配套：外部四个适配器的 filter 各加 `symbolic_math_tool` + `unit_tool` 两行。
